# Client-Server API Contract

이 문서는 구조 개편 후 클라이언트와 서버가 어떤 책임으로 API를 사용할지 정리한다.

## 기본 원칙

1. 클라이언트는 실시간 경험을 담당한다.
2. 서버는 공식 기록과 최종 AI 처리 결과를 담당한다.
3. 클라이언트가 만든 live 결과는 보조 정보다.
4. 서버가 만든 transcript, event, report가 공식 결과다.

## 클라이언트가 서버에 보내는 것

- 세션 생성 요청
- 세션 종료 요청
- 녹음 파일 업로드
- 필요 시 최종 transcript 업로드
- 회의록 생성 요청

## 서버가 클라이언트에 주는 것

- 세션 개요
- 저장된 이벤트 목록
- 회의록 생성 상태
- 최종 회의록 메타데이터

## 당장 유지하는 기존 흐름

- 로컬 MVP 기준 API는 계속 유지한다.
- 구조 개편 브랜치에서는 같은 의미의 계약을 `server/` 기준으로 재정리한다.

## 이후 확장 예정

- `workspace`
- `project`
- `participant`
- `recording`
- `transcript`
- `attachment`

이 확장 도메인이 추가되면 API도 세션 단독 구조에서 프로젝트/조직 문맥 구조로 이동한다.

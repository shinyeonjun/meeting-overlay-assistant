# 설계 메모

이 문서는 현재 CAPS의 코어 설계 원칙만 간단히 정리한다.

## 핵심 원칙

1. `overlay`와 `web workspace`의 역할을 분리한다.
2. 정본 저장소는 `PostgreSQL`이다.
3. retrieval과 memory는 `PostgreSQL + pgvector` 계층으로 붙인다.
4. 비동기 후처리는 `Redis + worker`로 분리한다.
5. 녹음, transcript, report는 DB 본문 대신 artifact 경로로 관리한다.

## 역할 분리

- `client/overlay`
  - 회의 중 HUD
  - 빠른 세션 생성, 시작/종료, 라이브 자막, 상태 표시
- `client/web`
  - 회의 후 중심 workspace
  - history, report, retrieval, assistant
- `server`
  - control API, live runtime, worker로 분리 가능한 구조

## 데이터 원칙

- 운영 DB: `PostgreSQL`
- 테스트 DB: `TEST_POSTGRESQL_DSN` 기반 별도 격리 PostgreSQL
- retrieval: `pgvector`
- queue / wake-up / lock: `Redis`
- artifact: raw audio, transcript, markdown, pdf

## 도메인 정리

- 사용자 로그인 키는 `login_id`
- 권한 키는 `workspace_members.workspace_role`
- 세션 입력 키는 `primary_input_source`, `actual_active_sources`
- 리포트는 `report_generation_job` 상태로 추적
- history는 retrieval brief까지 포함하는 read model

## 현재 방향

- `overlay`: Quick Start + Live Assist
- `web`: Review + Organize + Ask
- `server`: PostgreSQL only
- `sqlite`: active 코드에서 제거

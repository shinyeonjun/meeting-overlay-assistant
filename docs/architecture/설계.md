# 설계 메모

이 문서는 현재 코드베이스가 어떤 판단 위에 서 있는지 설명한다.  
기능 목록이 아니라 “왜 이렇게 설계했는가”를 정리한다.

## 1. 실시간과 최종 문서를 분리한 이유

초기에는 스트리밍 STT 결과를 그대로 요약/리포트에 연결하려고 했지만, 다음 문제가 반복됐다.

- partial 흔들림
- final 지연
- segment 경계 부정확
- 이벤트 오탐과 중복
- 리포트 근거 문장 신뢰도 하락

이 때문에 현재는 다음처럼 역할을 분리한다.

- **실시간 경로**
  - 자막 표시
  - 질문 중심 보조 인사이트
  - 사용자 즉시성 우선
- **리포트 경로**
  - 저장된 세션 오디오 사용
  - 고정밀 STT 재전사
  - 세션 종료 후 수동 생성
  - 최종 문서 신뢰도 우선

## 2. 자동 리포트 생성을 끈 이유

자동 생성은 편하지만 현재 품질 수준에서는 다음 문제가 있었다.

- 세션 종료 직후 사용자가 이벤트를 검토할 시간이 없음
- 결과가 틀렸을 때 바로 “공식 결과물”처럼 보임
- 생성 실패와 종료 실패가 한 경로에 묶여 장애 구분이 어려움

그래서 현재 정책은:

- `세션 종료`는 종료만 한다.
- `리포트 생성`은 세션 탭에서 사용자가 직접 누른다.

이 방식이 지금 단계의 신뢰도와 UX에 더 맞다.

## 3. 실시간 인사이트를 질문만 남긴 이유

결정 사항, 액션 아이템, 리스크는 실시간 경로에서 precision이 낮았다.
질문은 상대적으로 문장 형태가 명확해 분류 정확도가 높았다.

그래서 현재 정책은:

- live 이벤트: `question`만
- 최종 리포트 분석: `question / decision / action_item / risk`

즉 실시간과 최종 문서가 같은 분류 기준을 반드시 공유하지 않는다.

## 4. 이벤트 상태를 내부에만 유지하는 이유

사용자에게 `confirmed`, `candidate`, `resolved` 같은 상태를 그대로 노출하면 오히려 사용성이 나빠진다.
하지만 내부적으로는 다음이 필요하다.

- 리포트 반영 규칙
- 이력 추적
- 수정/재생성
- 상태 전이 검증

그래서 UI는 단순화하고, DB와 서비스 계층은 상태를 유지한다.

## 5. 산출물 폴더를 세션별로 나눈 이유

기존에는 `reports/` 아래 파일이 평평하게 쌓였다. 이 방식은 아래 문제가 있었다.

- 세션 단위로 결과 찾기 어려움
- 중간 산출물과 최종 산출물 혼재
- 버전 관리 가시성 부족

현재 구조:

```text
backend/data/reports/{session_id}/
  markdown.v1.md
  pdf.v1.pdf
  artifacts/
    *.transcript.md
    *.analysis.json
```

이 구조는 디버깅과 결과 정리에 모두 유리하다.

## 6. STT 전략을 하이브리드로 둔 이유

현재 운영 판단은 아래와 같다.

- `Sherpa`: 빠른 live partial
- `Faster-Whisper`: archive-final 및 리포트용 고정밀 STT

이유:
- Whisper 단독 live는 첫 final 지연이 크다.
- 경량 streaming 모델 단독은 최종 품질이 부족하다.
- 따라서 “회의 중 빠름”과 “회의 후 정확함”을 분리하는 편이 현실적이다.

## 7. 왜 LangChain / LangGraph를 핵심 경로에 도입하지 않았는가

비교 실험 결과, 이벤트 추출 도메인에서는 기존 prompt-based 방식이 더 나았다.

핵심 이유:
- 현재 병목은 프레임워크가 아니라 분류 precision
- 짧은 이벤트 추출은 후처리와 도메인 규칙이 더 중요
- 실시간 STT 경로는 프레임워크 이득보다 복잡도 증가가 큼

따라서 현재는:
- 이벤트 추출: 기존 Ollama prompt-based 유지
- LangChain: 도입 보류
- LangGraph: 도입 보류

## 8. 앞으로 바뀔 가능성이 있는 지점

다음은 향후 변경 가능성이 높은 영역이다.

### 리포트 분석 품질

- 중복 제거 강화
- 메타 문장 제거
- 섹션별 생성 규칙 정교화

### STT 계층 구조

- `partial / fast-final / archive-final`
- `early_eou / final_eou`
- `stability` 기반 UI 렌더

### 리포트 워크플로우

현재는 수동 생성이 기준이지만, 품질이 충분히 올라가면 다시 자동 생성이나 초안/확정 흐름을 검토할 수 있다.

# 설계 메모

이 문서는 현재 CAPS가 어떤 설계 판단 위에 서 있는지 설명합니다.  
기능 목록을 나열하기보다, 왜 지금 구조를 선택했는지와 어떤 축을 남기고 어떤 축을 제거했는지를 기록합니다.

현재 공식 기준 경로는 `server / client / shared / deploy`이고, `backend / frontend`는 과거 MVP 참조본입니다.

## 1. 핵심 설계 원칙

1. 실시간 보조 흐름과 회의 후 정식 문서 흐름을 분리한다.
2. 정본 데이터는 `PostgreSQL`에 둔다.
3. retrieval / memory는 `PostgreSQL + pgvector` 계층으로 붙인다.
4. 클라이언트는 캡처와 즉시 반응, 서버는 정본과 분석을 맡는다.
5. vertical slice 단위로 끝까지 연결하고, 최소 UI까지 붙여서 검증한다.

## 2. 실시간 흐름과 후처리 흐름을 분리한 이유

초기에는 실시간 STT 결과를 바로 리포트와 요약으로 연결하려 했지만, 아래 문제가 반복됐습니다.

- partial / final segment 흔들림
- 발화 경계 불안정
- 이벤트 중복과 누락
- 리포트 품질 불안정

그래서 현재는 역할을 분리했습니다.

- 실시간 흐름: live caption, question 중심 보조, 진행 중 인사이트
- 후처리 흐름: 전체 오디오 기반 transcript, 정식 리포트, retrieval 적재

즉 현재 설계는 `실시간 보조 + 회의 후 정식 문서화`입니다.

## 3. 리포트 생성 플로우

리포트는 세션 종료 후 후처리 파이프라인으로 만듭니다.

1. 세션 종료
2. 녹음 파일 확정
3. `report_generation_job` 생성
4. full STT transcript 생성
5. transcript 기반 markdown / pdf 리포트 생성
6. report와 knowledge 계층 적재

현재 리포트는 수동 생성보다 이 파이프라인을 기본 경로로 봅니다.

## 4. 실시간 인사이트의 범위

실시간 경로는 회의 중 즉시 반응이 중요한 축만 다룹니다.  
특히 `utterances`는 실시간 STT 발화 원재료이고, `overlay_events`는 그 발화를 근거로 만든 구조화 이벤트입니다.

이벤트 타입은 현재 아래 다섯 개입니다.

- `topic`
- `question`
- `decision`
- `action_item`
- `risk`

라이브에서는 `question` 중심으로 반응하고, 나머지 이벤트는 후처리 리포트와 history에서 더 안정적으로 소비합니다.

## 5. Context 모델을 먼저 세운 이유

이 제품은 단순 회의 녹음기가 아니라, 회의를 `Account / Contact / ContextThread`에 연결해 누적하는 구조를 지향합니다.

그래서 세션만 저장하지 않고 아래 맥락을 함께 들고 갑니다.

- 어느 회사(Account)와 관련된 회의인지
- 누구(Contact)와 관련된 회의인지
- 어떤 업무 흐름(ContextThread)에 속한 회의인지

이 구조 덕분에 history와 retrieval이 단순 로그가 아니라 “맥락 기반 회상”이 됩니다.

## 6. DB 설계 기준

현재 DB 설계는 아래 기준으로 정리됐습니다.

- 개발 기본 DB는 `PostgreSQL`
- 테스트 격리와 레거시 마이그레이션 호환은 `SQLite`
- retrieval / memory는 `pgvector`
- 로그인 식별자는 `login_id`
- 권한 기준은 `workspace_members.workspace_role`
- 세션 입력 정본은 `primary_input_source`, `actual_active_sources`

정리해서 제거한 축:

- `users.email`
- `users.role`
- `sessions.source`
- `sessions.participants_json`
- `screen_contexts`
- `overlay_events.source_screen_id`
- 이벤트 레거시 필드 (`priority`, `assignee`, `due_date`, `topic_group`)
- `reports.snapshot_markdown`
- `report_audit_logs`
- `knowledge_documents.search_text`

## 7. 현재 저장소 계층 방향

현재 목표 저장소 구조는 아래와 같습니다.

- `PostgreSQL`: 정본 비즈니스 데이터
- `pgvector`: retrieval / memory
- artifact storage: recording, transcript, markdown, pdf
- `SQLite`: 테스트 격리와 구버전 데이터 이행용

즉 운영 기준 저장소는 `PostgreSQL + pgvector`입니다.

## 8. 클라이언트 / 서버 분리 원칙

현재 아키텍처는 `capture near client, reason near server` 원칙을 따릅니다.

- `client`: 캡처, overlay UI, 즉시 상태 반응
- `server`: STT orchestration, 이벤트 추출, 리포트 생성, retrieval, 권한, 공유

이 구조 덕분에 Tauri overlay가 얇게 반응하고, 정본과 분석은 서버에서 일관되게 관리됩니다.

## 9. 운영 도구를 먼저 만든 이유

로컬/온프렘 환경에서는 제품 기능만큼 운영 안정성이 중요합니다.  
그래서 아래 관리 도구를 같이 유지합니다.

- `dashboard`
- `doctor`
- `logs`
- `settings`
- `profiles`

이 도구들은 제품 UX가 아니라 서버 운영 안정성과 진단을 위한 축입니다.

## 10. 현재 설계 요약

1. 실시간 보조와 정식 문서화를 분리했다.
2. 정본은 PostgreSQL, retrieval은 pgvector로 정리했다.
3. 세션을 Account / Contact / ContextThread 맥락 안에 넣었다.
4. 리포트는 후처리 파이프라인과 job 상태 기준으로 관리한다.
5. history는 read-model이고 retrieval brief까지 소비한다.
6. 클라이언트는 얇게, 서버는 정본과 분석을 맡는다.

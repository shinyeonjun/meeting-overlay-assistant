# 시스템 설계서

> 프로젝트명: 로컬 AI 회의 오버레이 어시스턴트  
> 문서 목적: `회의 중 지원`과 `회의 후 문서화` 두 기능을 중심으로 전체 시스템 구조를 정의한다.  
> 최종 갱신: 2026-03-05
> 최종 갱신: 2026-03-05

---

## 1. 시스템 한 줄 정의

```text
오디오와 화면을 계속 관찰하면서 회의의 핵심 이벤트를 추출해
회의 중에는 오버레이 카드로 보여주고,
회의 후에는 구조화된 문서로 정리한다.
```

---

## 2. 제품 관점에서의 두 기능

### 2.1 회의 중 기능

- 현재 회의에서 중요한 질문, 결정, 액션 아이템, 리스크를 빠르게 보여준다.
- 회의 진행을 방해하지 않는 선에서 현재 필요한 정보만 노출한다.
- 속도가 정확도보다 약간 더 중요하다.

### 2.2 회의 후 기능

- 회의 전체 흐름을 다시 묶어 문서로 정리한다.
- 질문과 답변, 결정사항, 액션 아이템, 리스크를 체계화한다.
- 정확도가 속도보다 더 중요하다.

### 2.3 핵심 설계 원칙

```text
같은 입력 데이터를 받되,
회의 중 기능과 회의 후 기능은 서로 다른 출력 목표를 가진다.

회의 중 기능은 빠른 이벤트 추출을,
회의 후 기능은 정확한 재구성과 문서 생성을 담당한다.
```

## 4. 입력과 출력

### 입력

1. 오디오 입력
- 시스템 오디오 (`backend WebSocket`)
- 마이크 입력 (기본: `Web Speech API`, fallback: `backend WebSocket`)

2. 화면 입력
- 현재 화면 캡처
- OCR 추출 텍스트

3. 시스템 상태 입력
- 세션 시작 / 종료
- 회의 모드 / 발표 모드

### 출력

1. 회의 중 출력
- `topic_update`
- `question_detected`
- `question_resolved`
- `decision_made`
- `action_item_created`
- `risk_detected`
- `live_context_update`

2. 회의 후 출력
- 전체 요약
- 주제 흐름
- 질문과 답변
- 결정사항
- 액션 아이템
- 리스크 및 후속 확인사항
- Markdown / HTML / PDF 문서

## 5. 회의 중 처리 플로우

```text
system_audio chunk 수신
    |
    v
VAD로 발화 구간 분리
    |
    v
STT로 발화 텍스트 생성
    |
    +--> 최근 화면 OCR 텍스트 결합
    |
    v
실시간 이벤트 추출
    |
    +--> 질문인가?
    +--> 결정사항인가?
    +--> 액션 아이템인가?
    +--> 리스크인가?
    +--> 현재 주제를 갱신해야 하는가?
    |
    v
상태 저장소 업데이트
    |
    v
오버레이 UI로 이벤트 푸시
```

### 5.1 소스별 입력 시퀀스 (축약)

#### `mic` (기본: Web Speech API)

```text
frontend live-connection
  -> /api/v1/ws/dev-text 연결
  -> Web Speech API 시작

interim(부분결과): 프론트 즉시 렌더링
final(확정결과): dev-text WebSocket 전송 -> backend 이벤트 추출
```

```text
fallback:
Web Speech API 오류/미지원
  -> /api/v1/ws/audio 전환
  -> backend STT(faster_whisper_streaming) 처리
```

#### `system_audio` (로컬 하이브리드 STT)

```text
Tauri/python 캡처
  -> /api/v1/ws/audio PCM 전송
  -> partial=sherpa, final=faster-whisper
  -> 오버레이 반영 + 이벤트 추출 연동
```

#### `mic_and_audio` (듀얼 입력)

```text
mic(Web Speech + /dev-text) 연결
+ system_audio(/audio PCM) 연결
-> 프론트 통합 렌더링
```

### 회의 중 처리 원칙

- 자막 전체를 그대로 노출하지 않는다.
- 발화를 카드 단위 이벤트로 구조화한다.
- 완벽한 확정보다 빠른 후보 제시를 우선할 수 있다.
- UI는 현재 중요한 정보만 보여준다.
- 이벤트는 생성 후 업데이트될 수 있다.

## 6. 회의 후 문서화 플로우

```text
세션 종료
    |
    v
누적 utterances / screen_contexts / overlay_events 조회
    |
    v
이벤트 중복 제거 및 상태 정리
    |
    v
주제별 그룹화
    |
    +--> 질문 / 답변 정리
    +--> 결정사항 정리
    +--> 액션 아이템 정리
    +--> 리스크 정리
    |
    v
최종 문서 생성
    |
    +--> Markdown
    +--> HTML
    +--> PDF
```

### 회의 후 처리 원칙

- 실시간 후보 이벤트를 그대로 복붙하지 않는다.
- 회의 전체 흐름을 다시 보고 정리한다.
- 동일 이벤트의 중복 감지를 허용하고, 문서화 단계에서 정제한다.
- 문서는 공유 가능한 수준의 구조와 문장 품질을 목표로 한다.

## 7. 회의 이벤트 모델

### 이벤트 종류

| 이벤트 | 의미 | 회의 중 역할 | 회의 후 역할 |
|---|---|---|---|
| `topic` | 현재 논의 중인 주제 | 상단 주제 표시 | 주제 흐름 정리 |
| `question` | 아직 답이 필요한 질문 | 미해결 질문 추적 | 질문 / 답변 섹션 |
| `decision` | 확정된 회의 결과 | 결정 카드 노출 | 결정사항 섹션 |
| `action_item` | 담당자와 해야 할 일 | 액션 카드 노출 | 후속 작업 표 |
| `risk` | 이슈, 장애, 주의사항 | 리스크 경고 | 리스크 및 확인사항 |
| `context` | 최근 맥락 정보 | 보조 맥락 노출 | 참고 정보 |

### 질문 상태 전이

```text
[감지됨]
   |
   v
[열림]
   |
   +--> 답변이 붙음 ------> [해결됨]
   |
   +--> 회의 종료 -------> [미해결]
```

### 액션 아이템 상태 전이

```text
[후보]
   |
   +--> 담당자 확인
   +--> 작업 확인
   +--> 기한 확인
   |
   v
[확정]
   |
   +--> 변경 발화 수신 ---> [수정됨]
   |
   +--> 회의 종료 -------> [문서 반영]
```

### 결정사항 상태 전이

```text
[후보]
   |
   +--> 확정 표현 감지
   |
   v
[확정]
   |
   +--> 뒤집는 발화 수신 --> [수정됨]
   |
   +--> 회의 종료 --------> [문서 반영]
```

## 8. 오버레이 UI 설계

### Live Overlay 현재 구조

```text

|오버레이 셸 (투명, 풀스크린, Tauri 네이티브)        |
|                                                   |
|                       +---[ 워크스페이스 ]---+   |
|                       | 세션 | 이벤트 | 리포트 |   |
|                       |                   |   |
|                       | 세션 제목/상태      |   |
|                       | 현재 주제          |   |
|                       | 이벤트 카드        |   |
|                       +-------------------+   |
|                                                   |
| +---[ 캡션 박스 ]--------------------------+ [⋮] |
| | 화자: 현재 발화 텍스트 █                |     |
```

### UI 원칙

- 캡션 박스에는 현재 발화 + 이전 피드 + 이벤트 피드를 보여준다.
- 우측 워크스페이스는 3개 탭(세션/이벤트/리포트)으로 구성된다.
- 본체 외 켄릭은 투과되어 하위 앱으로 전달된다.
- 캡션/워크스페이스/FAB 모두 드래그 이동 가능하다.
- 접기/펼치기, 탭 전환에 CSS 애니메이션 적용.

## 9. 회의 후 문서 설계

### 문서 구조

```text
회의 제목
회의 시간
참여 맥락 요약

1. 전체 요약
2. 주제 흐름
3. 질문과 답변
4. 결정사항
5. 액션 아이템
6. 리스크 및 후속 확인사항
7. 주요 화면 맥락
```

### 예시 구조

```markdown
# 2026-03-02 주간 개발 회의

## 1. 전체 요약
- 로그인 오류 대응 방안을 논의했다.
- 이번 배포 범위에서 수정 제외가 결정되었다.

## 2. 주제 흐름
1. 로그인 오류 원인 분석
2. 브라우저별 재현 여부
3. 배포 범위 결정

## 3. 질문과 답변
- 질문: 사파리에서만 재현되는가
  - 답변: 네, 현재는 사파리에서만 재현된다.

## 4. 결정사항
- 이번 배포에서는 로그인 수정 제외

## 5. 액션 아이템
- 민수: 수정안 정리, 금요일까지

## 6. 리스크
- 쿠키 정책으로 브라우저별 동작 차이가 존재
```

### 문서화 원칙

- 회의 중 UI에 보인 카드와 문서 항목이 자연스럽게 이어져야 한다.
- 문서에는 카드의 근거가 되는 발화 / 화면 맥락을 참조할 수 있어야 한다.
- 문서는 단순 요약이 아니라 실행 가능한 결과물이어야 한다.

## 11. 실시간성과 정확도의 균형

### 설계 원칙

- 회의 중 기능은 2~5초 수준의 짧은 분석 지연을 허용한다.
- 회의 후 기능은 세션 종료 뒤 추가 정리 시간을 허용한다.
- 실시간 카드와 최종 문서가 완전히 같은 문장일 필요는 없다.
- 대신 두 결과가 같은 사실 구조를 공유해야 한다.

### 이유

- 완전 실시간 자막은 STT 품질과 지연 문제가 크다.
- 회의 중에는 빠른 이벤트 추출이 더 중요하다.
- 회의 후에는 빠르기보다 문서 품질과 누락 방지가 더 중요하다.

## 12. GPU/NPU 역할 분리

### 현재 기준

- 실시간 STT 주 경로: GPU (`faster-whisper`, `sherpa` fast partial)
- NPU: 저전력 상시 감지/연구 트랙

### NPU 우선 적용 후보

1. always-on 음성 감지 전단
2. 경량 분류/태깅 보조

### 원칙

```text
핵심 가치는
"회의 중에는 중요한 것을 보여주고,
회의 후에는 문서로 완성하는 것"이다.

현 시점 실시간 STT는 GPU가 우세하고,
NPU는 저전력 상시 감지/연구 트랙에 집중한다.
```

## 13. 기술 리스크

| 리스크 | 영향 | 대응 |
|---|---|---|
| STT 품질 부족 | 질문 / 액션 추출 오류 | 이벤트 추출 규칙과 UI 보완 |
| 화면 OCR 오탐 | 현재 주제 혼선 | OCR은 보조 신호로만 사용 |
| NPU 경로 불안정 | 성능 시연 실패 | GPU 주 경로 유지 + NPU 실험 분리 |
| 오버레이 과부하 | 사용자 집중도 저하 | 카드 수 제한, 중요도 기반 노출 |
| 실시간 결과와 문서 불일치 | 제품 신뢰도 저하 | 공통 이벤트 저장소 기반 재구성 |

## 14. 핵심 설계 요약

```text
입력은 계속 받는다.
회의 중에는 빠르게 구조화한다.

같은 데이터를 쓰되,
```

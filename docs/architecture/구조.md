# 시스템 구조

## 개요

이 프로젝트는 로컬 환경에서 회의 음성을 처리해 두 가지 결과를 만든다.

1. **실시간 경로**
   - 오버레이 자막
   - 질문 중심 실시간 인사이트
2. **최종 리포트 경로**
   - 세션 종료 후 수동 생성
   - 저장된 녹음 기반 고정밀 STT
   - Markdown / PDF 리포트

핵심 설계는 `live는 빠름`, `report는 정확함`이다.  
실시간 자막과 최종 문서 품질을 같은 파이프라인으로 해결하려 하지 않고, 목적에 맞게 분리한다.

## 상위 구성

### 프론트엔드

- Tauri + Vite 기반 오버레이 앱
- 역할
  - readiness 확인
  - 세션 시작/종료
  - 실시간 자막 표시
  - 질문 이벤트 검토
  - 리포트 수동 생성

핵심 경로:
- `frontend/overlay/src/controllers/session-controller.js`
- `frontend/overlay/src/controllers/live/live-connection.js`
- `frontend/overlay/src/controllers/live/live-payload-handler.js`
- `frontend/overlay/src/controllers/live/live-caption-renderer.js`
- `frontend/overlay/src/controllers/events-controller.js`
- `frontend/overlay/src/controllers/report-controller.js`

### 백엔드

- FastAPI 기반 HTTP/WebSocket 서버
- 역할
  - 세션 관리
  - 오디오 입력 수신
  - STT 파이프라인 실행
  - 이벤트 생성/보정
  - 리포트 생성/조회
  - SQLite 저장

핵심 진입점:
- `backend/app/main.py`
- `backend/app/api/http/routes/`
- `backend/app/api/http/dependencies.py`

### 저장소

- SQLite: 세션, 발화, 이벤트, 리포트 메타
- 파일 시스템:
  - 리포트 산출물
  - transcript / analysis artifact
  - 세션 녹음 파일

## 처리 흐름

### 1. 세션 준비

1. 프론트가 `runtime/readiness`를 조회한다.
2. 백엔드는 STT preload 상태를 노출한다.
3. 프론트는 `ready` 상태가 되면 시작 버튼을 활성화한다.

이 단계는 “앱 준비”와 “실제 스트리밍 시작”을 분리하기 위한 것이다.

### 2. 실시간 오디오 경로

1. 세션 시작
2. 프론트가 Tauri 브리지를 통해 system audio 또는 mic 입력을 캡처
3. `/api/v1/ws/audio/{session_id}`로 PCM 스트림 전송
4. 백엔드 파이프라인이 partial / fast-final / archive-final 흐름으로 처리
5. 프론트는 가장 최근 live 자막과 누적 transcript를 따로 렌더링

핵심 모듈:
- `backend/app/services/audio/pipeline/audio_pipeline_service.py`
- `backend/app/services/audio/stt/hybrid_streaming_speech_to_text_service.py`
- `backend/app/services/audio/stt/sherpa_onnx_streaming_speech_to_text_service.py`
- `backend/app/services/audio/stt/faster_whisper_speech_to_text_service.py`
- `backend/app/services/audio/pipeline/stream_alignment_manager.py`

### 3. 이벤트 생성 경로

실시간 이벤트는 현재 질문만 노출한다.

1. live utterance 확정 또는 안정화 단계 도달
2. analyzer가 텍스트를 해석
3. 질문 이벤트만 저장/전달
4. 프론트 이벤트 탭에서 사용자가 `수정`, `처리 완료` 수행

핵심 모듈:
- `backend/app/services/analysis/analyzers/insight_pipeline_meeting_analyzer.py`
- `backend/app/services/analysis/analyzers/llm_based_meeting_analyzer.py`
- `backend/app/services/events/meeting_event_service.py`
- `backend/app/services/events/event_management_service.py`
- `backend/app/services/events/event_lifecycle_service.py`

### 4. 리포트 생성 경로

현재 리포트는 자동 생성하지 않는다. 사용자가 세션 종료 후 수동으로 생성한다.

1. 세션 종료
2. 세션 녹음 파일 경로 탐색
3. 고정밀 STT 수행
4. transcript / analysis artifact 생성
5. 구조화 리포트 템플릿 렌더링
6. Markdown 또는 PDF 저장

핵심 모듈:
- `backend/app/services/sessions/session_finalization_service.py`
- `backend/app/services/audio/io/session_recording.py`
- `backend/app/services/reports/audio/audio_postprocessing_service.py`
- `backend/app/services/reports/core/report_service.py`
- `backend/app/services/reports/refinement/structured_markdown_report_refiner.py`
- `backend/app/services/reports/composition/simple_pdf_writer.py`

## 현재 핵심 설계 원칙

### 1. 실시간 경로와 최종 문서 경로 분리

- 실시간: 빠른 반응과 읽기 쉬운 자막
- 최종 문서: 고정밀 transcript와 구조화 리포트

### 2. UI는 단순화, 내부 상태는 유지

사용자에게는 `수정`, `처리 완료` 같은 단순 액션만 노출하고, 내부 DB/서비스에서는 상태 전이와 버전 정보를 유지한다.

### 3. 산출물은 세션 폴더 기준

모든 리포트와 artifact는 `session_id` 폴더 아래에 모은다.  
최종 산출물과 중간 산출물을 `artifacts/`로 분리한다.

### 4. 개발용 경로와 제품 경로 분리

텍스트 입력용 WebSocket, 실험 스크립트, research 문서는 제품 경로와 분리해 둔다.

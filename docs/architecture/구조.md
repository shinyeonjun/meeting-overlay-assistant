# 시스템 구조

이 문서는 현재 CAPS 코드 기준의 공식 구조를 설명한다.

## 공식 경로

- `server/`
- `client/`
- `shared/`
- `deploy/`

`backend/`, `frontend/`는 예전 참조 경로다.

## 현재 구조 요약

### `client/overlay`

- Tauri 기반 데스크톱 클라이언트
- 로그인, 세션 시작/종료, live caption, 이벤트 보드, history, report 공유 UI 담당

### `server/app`

- FastAPI 중심 런타임
- 인증, 세션, participation, context, STT pipeline, 이벤트, 리포트, retrieval, 운영 모니터 담당

### `shared`

- 서버와 클라이언트가 같이 쓰는 계약, enum, schema

## 현재 핵심 흐름

### 세션

1. 사용자가 account/contact/thread를 선택한다.
2. draft 세션을 만든다.
3. `primary_input_source`로 세션을 시작한다.
4. `actual_active_sources`로 실제 활성 입력 소스를 추적한다.

### live 처리

1. 오디오가 WebSocket으로 들어온다.
2. STT가 `utterances`를 만든다.
3. 이벤트 추출이 `overlay_events`를 만든다.
4. UI는 live caption과 이벤트 보드를 갱신한다.

### post-meeting

1. 세션 종료
2. `report_generation_job` 생성
3. full STT / transcript
4. report 생성
5. report 공유
6. retrieval indexing

### history / retrieval

1. account/contact/thread 기준으로 history 조회
2. carry-over 계산
3. retrieval brief 결합

## 데이터 저장 기준

- 정본 DB: PostgreSQL
- retrieval 계층: PostgreSQL + pgvector
- 파일 artifact: `server/data/reports/{session_id}/`
- 테스트: SQLite 격리 인스턴스

## 현재 서비스 경계

- `auth`
- `session`
- `participation`
- `context`
- `events`
- `report`
- `history`
- `retrieval`
- `post_meeting`

## 현재 정책 메모

- 로그인 식별자는 `login_id`
- 권한은 `workspace_members.workspace_role` 기준
- 리포트 공유는 유지
- 별도 audit 로그 축은 제거
- `screen_contexts`와 OCR 축은 제거

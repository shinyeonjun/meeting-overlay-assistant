# 시스템 구조

현재 CAPS의 공식 구조는 `server / client / shared / deploy`다.

## 공식 경로

- `server/`
- `client/`
- `shared/`
- `deploy/`

`legacy/backend/`, `legacy/frontend/`는 과거 참조 경로다.

## 현재 구조 요약

### `client/overlay`

- Tauri 기반 회의 보조 HUD
- 빠른 세션 생성, 시작/종료, 라이브 자막, 상태 표시

### `client/web`

- 회의 후 중심 workspace
- history, report, retrieval, assistant UI

### `client/shared`

- overlay와 web이 같이 쓰는 API/auth/runtime 코드

### `server/app`

- FastAPI 기반 서버
- 인증, 세션, context, event, report, retrieval, worker orchestration 담당

## 데이터 경로

1. 세션 생성
2. live input 수집
3. utterance / event 저장
4. 세션 종료
5. web workspace에서 리포트 생성 요청
6. worker가 transcript / report / indexing 처리
7. web이 history / retrieval / assistant로 소비

## 저장소 기준

- 운영 DB: PostgreSQL
- 테스트 DB: 별도 PostgreSQL
- retrieval 계층: PostgreSQL + pgvector
- queue: Redis
- artifact: `server/data/reports/{session_id}/...`

# MVP 개발 백로그

> 문서 목적: 요구사항과 설계를 실제 구현 가능한 작업 단위로 분해하고, `bottom-up` 구현 전략 기준으로 우선순위와 완료 기준을 정의한다.  
> 상태: 아카이브(초기 백로그 기록). 현재 운영 기준은 `계획.md`, `설계.md`를 우선 참조한다.

---

## 1. 백로그 운영 원칙

- MVP는 `회의 중 기능`과 `회의 후 기능`의 핵심 연결만 먼저 완성한다.
- 구현은 `DB / 모델 -> 저장 / 조회 -> 도메인 서비스 -> 인터페이스 -> UI` 순서를 따른다.
- 상위 화면보다 하위 데이터 구조와 상태 일관성을 먼저 고정한다.
- 각 작업은 가능한 한 독립적으로 검증 가능해야 한다.
- “좋아 보이는 기능”보다 “데모에서 끊기지 않는 기능”을 우선한다.

---

## 2. 왜 bottom-up으로 가는가

이 프로젝트의 핵심은 화면보다 파이프라인이다.

```text
세션
  -> 발화
  -> 이벤트
  -> 오버레이
  -> 문서
```

이 흐름에서 앞단 저장 구조가 흔들리면 뒤쪽 UI와 문서화가 모두 다시 흔들린다.  
따라서 MVP는 아래 원칙으로 구현한다.

1. 먼저 저장 구조를 고정한다.
2. 그 다음 저장 구조를 읽고 쓰는 서비스를 만든다.
3. 그 다음 API / WebSocket 같은 인터페이스를 붙인다.
4. UI는 마지막에 연결한다.

---

## 3. MVP 완료 정의

MVP는 아래 조건을 만족하면 완료로 본다.

1. 세션 시작 / 종료가 된다.
2. 오디오 입력을 받아 발화 단위로 처리할 수 있다.
3. 질문, 결정, 액션 아이템, 현재 주제를 실시간에 가깝게 오버레이에 표시할 수 있다.
4. 세션 종료 후 Markdown 문서가 생성된다.
5. 오버레이에 보인 핵심 정보와 최종 문서가 대체로 일관된다.

## 4. 레이어 기준 에픽 구성

### Epic A. 데이터 레이어

목표:
- 세션, 발화, 이벤트, 문서를 안정적으로 저장하는 구조를 만든다.

### Epic B. 저장 / 조회 레이어

목표:
- 데이터 레이어를 읽고 쓰는 저장 흐름을 만든다.

### Epic C. 도메인 서비스 레이어

목표:
- 입력 데이터를 회의 도메인 이벤트로 변환한다.

### Epic D. 인터페이스 레이어

목표:
- API와 WebSocket으로 외부 입력과 출력을 연결한다.

### Epic E. 프레젠테이션 레이어

목표:
- 회의 중 오버레이와 회의 후 결과 노출을 만든다.

### Epic F. 품질 / 운영 레이어

목표:
- fallback, 로그, 테스트, 데모 품질을 보강한다.

# 스프린트 1 작업 계획

> 문서 목적: MVP 1차 스프린트를 `bottom-up` 구현 방식에 맞춰 세분화하고, 작업 순서 / 산출물 / 검증 기준 / 리스크를 정의한다.  
> 상태: 아카이브(초기 스프린트 기록). 현재 운영 기준은 `계획.md`, `설계.md`를 우선 참조한다.

---

## 1. 스프린트 목표

스프린트 1의 목표는 아래 한 줄로 정리된다.

```text
DB와 저장 구조를 먼저 고정하고,
그 위에 발화 처리와 이벤트 저장까지 연결한다.
```

이번 스프린트에서 실제로 살아 있어야 하는 최소 흐름은 다음과 같다.

```text
세션 저장
  -> utterance 저장
  -> question / decision / action_item 이벤트 저장
  -> API / WebSocket으로 연결 가능
```

즉, 스프린트 1은 `UI 이전`, `문서화 이전`, `도메인 저장 구조 우선` 스프린트다.

## 3. 스프린트 완료 정의

스프린트 1은 아래 조건을 만족하면 완료다.

1. 세션을 생성 / 종료할 수 있다.
2. WebSocket으로 오디오 입력을 받을 수 있다.
3. 발화 텍스트를 저장할 수 있다.
4. 질문 / 결정 / 액션 이벤트가 DB에 저장된다.
5. 저장 구조와 인터페이스가 이후 오버레이 / 문서화에 재사용 가능하다.

## 9. 스프린트 리스크

### 리스크 1. STT 연동이 예상보다 오래 걸릴 수 있음

대응:
- 초기에는 텍스트 입력 기반 검증 경로를 같이 둔다.

### 리스크 2. 발화 구간 분리가 불안정할 수 있음

대응:
- VAD를 완벽히 만들기보다 고정 길이 chunk 기반 임시 경로도 고려한다.

### 리스크 3. 이벤트 추출 정확도가 낮을 수 있음

대응:
- 규칙 기반 1차 구현부터 시작하고, 후보 이벤트 허용

## 11. 최종 정리

```text

"저장 구조 -> 도메인 처리 -> 인터페이스 연결"이 끊기지 않는
백엔드 하부 구조를 만드는 것"이다.
```

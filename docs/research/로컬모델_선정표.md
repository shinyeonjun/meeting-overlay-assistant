# 로컬 모델 선정표

## 목적

이 문서는 `AMD Ryzen AI 9 365 + NPU + RTX 5070 Laptop GPU` 환경에서  
회의 보조 시스템에 사용할 로컬 STT / LLM 모델의 우선순위를 정리합니다.

핵심 원칙은 다음과 같습니다.

- 회의 중에는 `실시간성` 우선
- 회의 후에는 `정확도와 문서 품질` 우선
- 입력 소스별로 최적 경로를 분리 (`mic`와 `system_audio`를 동일 경로로 강제하지 않음)
- 직접 변환보다 `공식 최적화 경로`를 먼저 사용

## 현재 결론

### STT 주력 경로

- `mic`: `faster_whisper_streaming` (현재 운영)
- `system_audio`: `hybrid_local_streaming` (partial=sherpa, final=faster-whisper)
- 실험/연구 경로: `amd/whisper-small-onnx-npu`

### LLM 주력 경로

- 기본값: `qwen2.5:3b-instruct` via `Ollama`
- 고품질 대안: `qwen2.5:7b`
- 경량 대안: `phi4-mini:3.8b`

## STT 벤치마크 결론

동일 샘플(`test_16k_mono_15s.wav`) 기준 결과:

| 항목 | faster-whisper GPU | AMD Whisper NPU |
|---|---:|---:|
| WER | 0.1714 | 0.3429 |
| CER | 0.1140 | 0.1754 |
| RTF | 0.2697 | 1.1494 |
| First Guarded Latency | 2.8243s | 14.3790s |
| Peak RSS | 약 468MB | 약 3.18GB |

현재 결과만 보면 `GPU가 속도, 정확도, 메모리 모두 우세`합니다.

따라서 현 단계에서 제품 기본 STT는 `faster-whisper GPU`로 두고,
NPU는 `저전력 상시 감지`와 `향후 최적화 연구`를 위한 트랙으로 유지합니다.

추가로 `system_audio` 실시간 후보 5개 모델 비교를 완료했습니다.

| 모델/엔진 | first_partial | first_final | RTF(total) | 판정 |
|---|---:|---:|---:|---|
| `faster_whisper_streaming` | 4.7097s | 5.2344s | 2.9344 | 실시간 partial 엔진으로는 부적합 |
| `moonshine_streaming` | 1.07s | 2.51s | - | 한국어 정확도 부족 (탈락) |
| `simulstreaming` | 1.24s | 15.0s | 1.932 | 지연/정확도 모두 부적합 (탈락) |
| `sherpa_onnx_streaming` | 0.0622s | 0.9698s | 0.3401 | 속도 우수, 품질 추가 튜닝 필요 |
| `sensevoice_small_streaming` (CPU) | 0.2231s | 41.9867s | 3.8314 | 현 설정 기준 부적합 |

> 상세 수치와 원본 파일: [벤치마크_비교표.md](벤치마크_비교표.md)

## STT 후보 정리

| 구분 | 모델 | 추천도 | 실행 장치 | 용도 | 판단 |
|---|---|---|---|---|---|
| 주력(`mic`) | `faster_whisper_streaming` | ✅ 추천 | GPU/CPU | 마이크 실시간 자막 | 현재 기본값 |
| 주력(`system_audio-final`) | `deepdml/faster-whisper-large-v3-turbo-ct2` | ✅ 매우 추천 | GPU | final 보정 | 현재 기본값 |
| 운영(`mic`) | `Web Speech API` | 조건부 추천 | Browser | 저지연 마이크 경로 | 기본 경로(실패 시 backend fallback) |
| 연구 | `amd/whisper-small-onnx-npu` | 추천 | NPU | 저전력 / 비교 실험 | 연구 트랙 |
| 연구 | `amd/whisper-medium-onnx-npu` | 조건부 추천 | NPU | 정확도 개선 실험 | 다음 단계 후보 |
| 배치 | `openai/whisper-large-v3-turbo` | 추천 | GPU | 회의 후 재전사 | 후처리용 |
| 실시간 후보 | `sherpa-onnx-streaming-zipformer-korean-2024-06-16` | 조건부 추천 | CPU/GPU | system_audio partial | 하이브리드 fast 경로 |
| 실험 | `SenseVoiceSmall` | 보류 | CPU (현재) | 실시간 후보 평가 | final 지연 과대 |
| ❌ 탈락 | `moonshine/tiny-ko` | 비추천 | GPU/CPU | 초경량 스트리밍 | WER 0.86 → 한국어 정확도 부족 |
| ❌ 탈락 | `simulstreaming (large-v3-turbo.pt)` | 비추천 | GPU | sliding window 스트리밍 | RTF 1.93, WER 0.97 → 실시간성·정확도 모두 부족 |

> 상세 벤치마크 결과: [벤치마크_비교표.md](벤치마크_비교표.md)

## 실시간감 확보: Pseudo-Streaming (2026-03-04 추가)

moonshine/simulstreaming이 탈락한 이후, faster-whisper를 유지하면서 실시간감을 확보하는 전략으로 **pseudo-streaming partial/final** 방식을 채택했습니다.

| 계층 | 현황 |
|---|---|
| 마이크 실시간 | ✅ `Web Speech API` 기본 운영, `faster_whisper_streaming` fallback |
| 프론트엔드 | ✅ 구현 완료 (`segment_id` 기반 partial→final 교체) |
| 백엔드 | ✅ 하이브리드 연결 완료 (`partial=sherpa`, `final=faster-whisper` 경로 확보) |
| 튜닝 과제 | 🔧 `standalone_final` 축소, system_audio 품질 개선 |

## LLM 후보 정리

| 구분 | 모델 | 추천도 | 실행 장치 | 용도 |
|---|---|---|---|---|
| 기본 | `qwen2.5:3b-instruct` | 매우 추천 | GPU/CPU | 회의 후 문서화, 구조화 출력 |
| 대안 | `phi4-mini:3.8b` | 추천 | GPU/CPU | 경량 추론, reasoning 비교 |
| 품질 우선 | `qwen2.5:7b` | 추천 | GPU | 문서 품질 향상 |

## 하이브리드 전략

이 프로젝트의 현재 아키텍처는 `GPU와 NPU를 비교해 역할을 분리하는 것`이 핵심입니다.

| 역할 | 엔진 | 이유 |
|---|---|---|
| 실시간 STT 주력 | GPU (`faster-whisper`) | 속도, 정확도, 메모리 모두 우세 |
| 상시 감지 / 저전력 전단 | NPU | always-on 연구 가치 |
| 회의 후 요약 / 문서화 | 로컬 LLM (`Ollama`) | 구조화 문서 품질 확보 |

즉 포인트는 `NPU만 사용`이 아니라:

> GPU와 NPU를 실제 비교하고, 각 장치가 잘하는 역할에 맞춰 파이프라인을 재배치했다.

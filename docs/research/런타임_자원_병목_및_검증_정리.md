# CAPS 런타임 자원, 병목, 검증 정리

작성 시각: 2026-04-14 17:03:05 KST
재검증 시각: 2026-04-26 08:07 KST

## 목적

이 문서는 현재 CAPS 로컬 개발 환경에서 어떤 경로가 CPU, RAM, GPU를 사용하는지 정리하고, 실제 병목이 어디에 있는지 빠르게 판단하기 위한 운영 메모다.

핵심 질문은 아래와 같다.

- 어떤 경로가 GPU를 쓰는가
- 어떤 경로가 동시에 겹칠 때 병목이 생기는가
- 실제로 부족한 자원은 무엇인가
- 지금 단계에서 가장 효과가 큰 최적화는 무엇인가

## 한 줄 결론

현재 CAPS의 가장 큰 병목은 `CPU`나 `RAM`이 아니라 `8GB VRAM을 가진 dGPU`와 `GPU 동시성 관리`다.

구조상 아래 경로가 같은 시점에 겹치면 GPU 병목이 생길 수 있다.

- `faster-whisper` final STT
- `pyannote` diarization
- `Ollama gemma4:e4b` 기반 workspace summary

즉, 지금 우선순위는 더 가벼운 모델을 찾는 것보다 `live는 CPU`, `final은 GPU`, `배치 요약은 비실시간`으로 경로를 더 분리하는 것이다.

## 구조 개선 반영 상태 (2026-04-24)

아래 항목은 문서에서 구조적 비효율로 잡았고, 현재 코드에 1차 개선을 반영했다.

| 항목 | 이전 구조 | 현재 상태 |
| --- | --- | --- |
| post-processing worker 병렬 처리 | 기본 batch가 여러 job을 한 번에 claim할 수 있어 GPU-heavy 작업이 겹치기 쉬움 | heavy worker 기본 batch를 `1`로 낮춰 로컬 8GB VRAM 기준 직렬 처리에 가깝게 조정 |
| live 중 post-processing claim | live session이 있어도 post-processing job을 먼저 claim하고 처리 진입 가능 | running session이 있으면 post-processing job을 claim하지 않고 다음 polling으로 넘김 |
| 직접 post-processing 실행 | 수동/동기 실행 경로에서는 live와 겹칠 수 있음 | heavy stage 진입 전 running session 종료를 대기하고, timeout 후에만 기존 흐름으로 진행 |
| workspace summary 실행 | summary가 live/post-processing과 같은 타이밍에 GPU/RAM을 경쟁할 수 있음 | summary 실행 전 running session과 active post-processing을 대기하고, live 중에는 summary job claim도 보류 |
| post-processing과 summary 산출물 정합성 | 후처리 재생성 후 기존 summary/correction artifact가 남을 수 있음 | canonical transcript 재생성 시 correction/summary artifact를 함께 정리하고 실패 시 복구 |
| GPU-heavy stage 계측/제어 지점 | summary와 post-processing이 서로 다른 방식으로 실행됨 | `gpu_heavy_execution_gate` 경유 지점을 summary와 post-processing에 둘 다 둠 |
| mic live GPU 경합 | `mic`가 서버 `faster_whisper_streaming`일 때 live partial부터 GPU를 사용하고, 동시 session에서 backlog가 커짐 | 기본 mic 경로는 Web Speech API + `text_input`으로 두고, 서버 mic STT는 `MIC_SERVER_STT_FALLBACK_ENABLED=true`일 때만 허용. 기본 preload에서도 mic STT 제외 |
| post-processing 큰 덩어리 관측 | 전체 후처리 시간이 하나로만 보여 실제 병목 stage를 바로 특정하기 어려움 | `prepare`, `load_audio`, `diarize`, `stt`, `build`, `persist` stage 상태와 elapsed 로그를 추가 |
| post-processing 재시도 반복 비용 | 실패/재처리 시 `pyannote`와 `faster-whisper`를 다시 실행함 | `diarize`와 `stt` 산출물을 session artifact cache로 저장하고, 같은 녹음/hash/signature면 heavy stage를 재사용 |
| 실시간 질문/이벤트 노출 | live analyzer가 회의 중 이벤트를 만들고 overlay가 질문/이벤트 feed를 노출할 수 있음 | MVP 기본값은 `LIVE_ANALYZER_BACKEND=noop`, `LIVE_QUESTION_ANALYSIS_ENABLED=false`, `VITE_LIVE_EVENT_INSIGHTS_ENABLED=false`로 두고 회의 중 화면은 자막 중심으로 제한 |
| 후처리 LLM 과사용 | `POST_PROCESSING_ANALYZER_BACKEND=insight_pipeline`이면 utterance마다 `rule_based + llm` 분석을 수행 | MVP 기본값을 `rule_based`로 낮추고, 무거운 AI는 리포트/노트 인사이트/챗봇의 비실시간 옵션으로 격리 |

아직 남은 제약과 다음 개선 후보는 아래다.

1. `mic` 서버 STT fallback을 켜면 여전히 `faster_whisper_streaming`이라 partial 단계에서 GPU를 쓴다. 기본값은 꺼져 있으므로 예외 상황용으로만 둔다.
2. stage cache는 같은 session/녹음/hash/signature 기준으로만 동작한다. 녹음이 바뀌거나 backend 설정이 바뀌면 의도적으로 cache miss가 난다.
3. 첫 post-processing은 여전히 `pyannote`와 `faster-whisper` 비용을 그대로 낸다. cache는 실패/재처리 반복 비용을 줄이는 개선이지, 최초 처리 비용 제거는 아니다.
4. `gemma4:e4b`는 비실시간으로 분리했지만, 실행되면 여전히 Ollama RSS 약 `7.9 GB`급 부담이 있다.
5. NPU/AMD Whisper 경로는 실험 backend라 아직 운영 대체 경로로 확정하지 않았다.

### post-processing stage cache 적용 후 실측 (2026-04-24 08:16 KST)

같은 session과 같은 녹음 파일로 post-processing job을 2회 실행했다. cache key는 `recording_artifact_id`, 녹음 파일 `sha256`, 후처리 pipeline signature가 모두 맞아야 hit 된다.

| 구분 | 전체 | diarize | stt | GPU-heavy 실행 |
| --- | ---: | ---: | ---: | --- |
| 1회차 | `34.272s` | `11.645s` | `18.623s` | `diarize`, `stt` 둘 다 실행 |
| 2회차 | `0.578s` | `0.008s` | `0.029s` | 둘 다 cache hit, GPU gate 미진입 |

해석:

- 실패/재시도/수동 재생성처럼 같은 녹음을 다시 처리하는 케이스는 이제 heavy stage 반복 비용이 거의 사라진다.
- 녹음 파일 내용이 바뀌면 SHA-256 mismatch로 cache miss가 나고, `diarize`와 `stt`를 다시 실행한다. 이 동작은 테스트로 고정했다.
- 단, 새 session이나 다른 녹음은 cache 대상이 아니다. 일반적인 첫 처리 속도는 여전히 하드웨어와 모델 비용을 따른다.

### 2. 성능 재측정 결과

이번 수치는 이전 병렬 실행으로 오염된 측정을 제외하고, 가능한 한 단독 실행 기준으로 다시 잡았다.

#### audio post-processing

### 1. workspace summary는 현재 가장 비싼 경로다

`gemma4:e4b`를 실제 `workspace summary synthesis` 경로로 돌렸을 때 결과는 아래와 같았다.

핵심 해석:

- warm run이 cold run보다 빨라지지 않았다. 즉 이 머신에서는 로딩보다 `실제 추론 비용`이 더 큰 문제다.

### 2. diarization은 비싸지만, post-processing 전체보다 덜 비싸다

15초 mono WAV 기준 실측은 아래와 같았다.

| --- | ---: | ---: | ---: | ---: |
| `pyannote_worker` | `15s` | `15.202s` | `1.01x` | `1774.21 MB` |
| `audio post-processing` | `15s` | `67.632s` | `4.51x` | `3762.75 MB` |

추가로 같은 프로세스에서 `audio post-processing`을 두 번 연속 실행했을 때는 `30.424s -> 16.773s`로 줄었다.

핵심 해석:

- `pyannote` 단독도 무겁지만, 실제 병목은 diarization 하나보다 `STT + 후처리 파이프라인 전체`에서 더 크게 나타난다.
- `session post-processing`은 cold start 비용도 의미 있게 있다.
- 이 경로는 live와 경쟁시키기보다 `배치 전용`으로 두는 쪽이 맞다.

### 3. live runtime은 `mic`보다 `system_audio`가 더 효율적이다

현재 벤치는 `preview` 완료 기준이다. 즉 `final_emit_count=0` 상태에서 `preview latency`와 `chunk backlog`를 비교한 수치다.

| 소스 | worker | session | terminal p95 | coalesced chunks |
| --- | ---: | ---: | ---: | ---: |
| `mic` | 1 | 4 | `9520.57 ms` | `454` |
| `mic` | 2 | 4 | `10250.34 ms` | `454` |
| `system_audio` | 1 | 2 | `8950.27 ms` | `187` |
| `system_audio` | 2 | 2 | `8144.23 ms` | `184` |

핵심 해석:

- `mic / faster_whisper_streaming`은 worker를 `2`로 늘려도 동시성 구간에서 거의 개선되지 않았고, `sessions=4`에서는 오히려 악화됐다.
- `system_audio / hybrid_local_streaming`은 같은 조건에서 backlog가 더 낮고, `worker=2`일 때 terminal p95도 소폭 개선됐다.
- 즉 현재 프로파일 중에서는 `system_audio` 쪽이 훨씬 덜 비효율적이고, `mic` 쪽이 먼저 손볼 대상이다.

### 4. 이번 재검증으로 확정된 운영 판단

- 현재 머신의 최상위 병목은 여전히 `CPU`나 `RAM`보다 `8GB VRAM + GPU 동시성`이다.
- 하지만 이제는 추정이 아니라, `workspace summary`가 실제로 가장 비싼 경로라는 수치가 붙었다.
- 따라서 우선순위는 `모델 교체`보다 `GPU-heavy 경로의 동시 실행 차단`, `mic live의 CPU partial화`, `stage별 계측 로그 추가`다.

## 현재 부족한 자원과 원인

### 1순위: GPU VRAM

현재 dGPU VRAM은 `8151 MiB`다. 이 환경에서 아래 조합은 동시에 오래 유지되기 어렵다.

- `faster-whisper-large-v3`
- `pyannote`
- `gemma4:e4b`

즉 문제는 GPU가 없어서가 아니라 `GPU에 올라오는 작업이 동시에 너무 무겁고, 일부 모델은 CPU까지 같이 끌어쓴다`는 데 있다.

### 2순위: GPU 동시성 관리

현재 병목은 “모델 하나가 느리다”보다 아래 케이스에서 생길 가능성이 높다.

- live mic가 `faster_whisper_streaming`으로 GPU 사용
- 동시에 post-processing final STT가 GPU 사용
- 동시에 pyannote worker가 GPU 사용
- 동시에 note correction worker가 `gemma4:e4b` summary 실행

이 구조는 `단일 순간 utilization`보다 `겹치는 순간의 VRAM 경쟁 + CPU/GPU 혼합 모델의 간섭`이 문제다.

### 3순위: 프로세스 단위 중복 로딩

`faster-whisper`는 process-local model cache가 이미 있다. 즉 한 프로세스 안에서는 재로딩 낭비가 적다.

하지만 worker가 분리된 프로세스로 여러 개 뜨면 아래 문제가 다시 생긴다.

- 모델 warm-up 중복
- VRAM 상주 중복
- RAM 복사본 증가

즉 현재 병목은 `캐시 없음`이 아니라 `프로세스 단위 중복 로딩 가능성`이다.

### 아직 여유가 있는 자원

- CPU: 상대적으로 여유가 있다.
- RAM: 31GB라 아직 버틸 수 있다.
- iGPU/NPU: 활용 여지가 있으나 현재 경로에 거의 연결되어 있지 않다.

## 현재 구조의 장점

좋은 점도 분명히 있다.

- system audio live 경로는 이미 `CPU partial + GPU final`로 잘 나뉘어 있다.
- STT preload가 있어 live runtime cold start를 줄이는 방향으로 가고 있다.
- live question worker와 live event board는 MVP 기본값에서 비활성화되어 회의 중 자원 사용을 줄인다.
- report generation은 현재 `noop refiner`라 병목이 아니다.

즉 구조가 완전히 나쁜 건 아니고, `몇 군데 무거운 경로가 동시에 겹치는 설계`가 문제다.

## 바로 적용해볼 가치가 큰 개선안

### A. live는 CPU, final은 GPU 원칙을 더 강하게 적용

가장 먼저 검토할 것은 `mic` 경로도 system_audio처럼 `CPU partial + GPU final`로 옮길 수 있는지다.

현재 기준으로 가장 효율 좋은 방향은 아래다.

- live partial: `sherpa` 또는 다른 CPU/NPU 경로
- final STT: `faster-whisper` on GPU

이 구조가 live 체감과 final 품질을 둘 다 살리기 쉽다.

### B. summary는 live와 GPU를 경쟁하지 않게 하기

`workspace summary`는 정확도가 중요하지만 live-critical path는 아니다.

따라서 아래 중 하나가 맞다.

- live session 중에는 summary 실행을 뒤로 미루기
- note/reprocess 전용 배치 시간에만 summary 실행
- 요약 worker 실행 시 live GPU 부하가 낮을 때만 돌리기

### C. pyannote는 실시간이 아니라 post-processing 전용 유지

현재처럼 diarization을 batch/post 단계로 두는 게 맞다.
다만 이 worker가 note summary와 동시에 돌지 않게 스케줄링하는 것이 중요하다.

### D. 무거운 worker 개수는 늘리지 않기

현재 하드웨어에서는 아래 방향이 안전하다.

- `post-processing worker`: 1개
- `note correction/summary worker`: 1개
- `report worker`: 1개
- `live question worker`: MVP에서는 실행하지 않음

이 이상 병렬도를 올리면 처리량이 늘기보다 자원 경쟁만 커질 가능성이 높다.

### E. 실측 로그를 더 남기기

지금은 코드/설정상 병목에 더해, 일부 모델의 실측 경향까지 확인한 상태다. 다음 단계는 아래를 stage별로 자동 기록하는 것이다.

- 시작 시각
- 종료 시각
- elapsed
- 모델명
- backend
- cold start 여부
- `nvidia-smi` 기준 GPU memory used
- Python process working set

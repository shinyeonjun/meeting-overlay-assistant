# 이벤트 처리 정책 및 상태 전이 명세

> 문서 목적: 회의 중 이벤트가 어떻게 생성, 갱신, 종료되는지와 회의 후 문서화 단계에서 어떻게 정제되는지를 정의한다.

---

## 1. 문서 역할

이 문서는 아래 문제를 해결하기 위한 기준이다.

- 같은 질문이 여러 번 말되었을 때 새 이벤트로 볼지 기존 이벤트 업데이트로 볼지
- 액션 아이템 정보가 불완전할 때 어떤 상태로 저장할지
- 실시간 후보 이벤트와 최종 문서 이벤트를 어떻게 연결할지
- 이벤트가 언제 오버레이에 뜨고 언제 사라질지

즉, 이 문서는 `회의 중 빠른 구조화`와 `회의 후 정확한 정리` 사이의 중간 규칙을 담당한다.

---

## 2. 기본 원칙

### 원칙 1. 실시간 이벤트는 완성본이 아니다

- 회의 중 이벤트는 빠른 반응을 위해 후보 상태로 시작할 수 있다.
- 최종 문서에는 세션 종료 후 정제된 결과만 반영한다.

### 원칙 2. 같은 사실은 가급적 같은 이벤트로 유지한다

- 동일 질문, 동일 결정, 동일 액션 아이템은 새 이벤트 생성보다 기존 이벤트 업데이트를 우선한다.

### 원칙 3. 근거를 남긴다

- 모든 핵심 이벤트는 가능하면 발화 또는 화면 맥락 근거를 연결한다.

### 원칙 4. 오버레이와 문서는 같은 저장소를 공유한다

- 오버레이에 보인 이벤트와 최종 문서는 `overlay_events`를 공통 기준으로 사용한다.

---

## 3. 이벤트 공통 속성

모든 이벤트는 최소한 다음 속성을 가진다.

- `event_type`
- `title`
- `state`
- `priority`
- `topic_group`
- `created_at_ms`
- `updated_at_ms`
- `source_utterance_id` 또는 `source_screen_id`

추가 속성:

- 액션 아이템: `assignee`, `due_date`
- 질문 해결: `answer_summary`
- 문서화 단계: `normalized_title`, `merged_into`

---

## 4. 이벤트 생명주기

```text
감지
  |
  v
후보 생성
  |
  +--> 추가 근거 유입 -> 갱신
  |
  +--> 확정 표현 감지 -> 확정
  |
  +--> 회의 종료 -> 문서화 정제
  |
  +--> 더 이상 유효하지 않음 -> 종료 / 보관
```

---

## 5. 이벤트별 상태 전이

### 5.1 주제 이벤트

상태:
- `active`
- `replaced`
- `archived`

전이 규칙:
- 새 주제가 감지되면 기존 `active` 주제는 `replaced` 된다.
- 세션 종료 시 마지막 주제를 포함한 전체 주제 이력은 `archived` 처리 가능하다.

생성 조건:
- 최근 발화 클러스터가 기존 주제와 다르다고 판단될 때
- 화면 OCR 제목 또는 키워드가 주제 변경을 지지할 때

업데이트 조건:
- 같은 주제 내 세부 설명이 추가될 때 `title` 또는 `body`를 보강한다.

---

### 5.2 질문 이벤트

상태:
- `open`
- `answered`
- `closed`
- `unresolved`

생성 조건:
- 의문문 또는 확인 요청 표현이 감지될 때
- 명시적 질문 표현이 없더라도 답변을 요구하는 발화가 강하게 감지될 때

업데이트 조건:
- 같은 질문을 다시 언급하면 기존 이벤트의 `updated_at_ms`를 갱신한다.
- 답변 후보가 붙으면 `body` 또는 `answer_summary`를 보강한다.

종료 조건:
- 답변이 충분히 연결되면 `answered`
- 세션 종료 시 답변이 없으면 `unresolved`
- 문서 반영 후 `closed`

중복 판정 기준:
- `title` 의미가 유사함
- 같은 `topic_group`
- 최근 일정 시간 내 재등장

---

### 5.3 결정 이벤트

상태:
- `candidate`
- `confirmed`
- `updated`
- `closed`

생성 조건:
- “그렇게 하자”, “이번에는 제외”, “이걸로 결정” 같은 확정 표현이 감지될 때
- 아직 확정 어조가 약하면 `candidate`

업데이트 조건:
- 후속 발화에서 조건, 범위, 예외가 추가되면 `updated`
- 기존 결정을 뒤집는 발화가 나오면 같은 이벤트 수정 또는 새 이벤트 생성 여부를 검토한다.

종료 조건:
- 문서 반영 후 `closed`

분기 원칙:
- 기존 결정을 수정하는 정도면 같은 이벤트 업데이트
- 정반대 방향의 결정을 새로 내리면 새 이벤트 생성 후 이전 이벤트는 `updated` 표시

---

### 5.4 액션 아이템 이벤트

상태:
- `candidate`
- `confirmed`
- `updated`
- `closed`

생성 조건:
- 누가 무엇을 해야 한다는 후속 조치가 감지될 때
- 담당자 또는 기한이 불명확해도 우선 `candidate`로 생성 가능

업데이트 조건:
- 담당자 확정
- 작업 내용 구체화
- 기한 확정 또는 변경

종료 조건:
- 문서 반영 후 `closed`

데이터 보강 순서:
- 1차: 작업 내용
- 2차: 담당자
- 3차: 기한

오버레이 노출 규칙:
- `candidate`는 낮은 강조
- `confirmed`는 고정 카드

---

### 5.5 리스크 이벤트

상태:
- `active`
- `monitoring`
- `resolved`
- `closed`

생성 조건:
- 장애, 불확실성, 우려, 제약사항 발화가 감지될 때

업데이트 조건:
- 영향 범위 추가
- 대응 방안 추가
- 동일 리스크 재언급

종료 조건:
- 해결되면 `resolved`
- 문서 반영 후 `closed`

---

### 5.6 컨텍스트 이벤트

상태:
- `live`
- `expired`

용도:
- 최근 발화나 화면 맥락을 짧게 보여주는 보조 정보

규칙:
- 장기 보존보다 회의 중 보조 노출이 목적이다.
- 오래된 정보는 자동 `expired`

---

## 6. 중복 제거 및 병합 정책

### 6.1 실시간 단계

실시간 단계에서는 완전한 중복 제거보다 아래 원칙을 따른다.

- 같은 사실일 가능성이 높으면 기존 이벤트 업데이트를 우선
- 애매하면 새 이벤트를 생성하되 `candidate` 또는 낮은 priority로 둔다

### 6.2 회의 후 단계

세션 종료 후에는 아래 정책으로 병합한다.

1. 의미가 같은 질문 병합
2. 같은 결정의 반복 언급 병합
3. 같은 액션 아이템의 정보 보강 이력을 하나로 병합
4. 리스크 중복 문장 제거

### 6.3 병합 판단 요소

- 제목 의미 유사도
- 같은 주제 그룹 여부
- 시간적 근접성
- 근거 발화의 연속성
- 담당자 / 기한 / 키워드 일치

---

## 7. 우선순위 산정 정책

`priority`는 0~100 기준으로 가정한다.

### 기본 기준

- 질문: 70
- 결정: 85
- 액션 아이템: 90
- 리스크: 75
- 컨텍스트: 30
- 현재 주제: 별도 상단 고정

### 가중치 요소

- 확정 표현 포함: +10
- 담당자 / 기한 포함: +10
- 같은 내용 반복 언급: +5
- 화면 OCR이 근거를 보강: +5
- 불확실성 높음: -10

---

## 8. 오버레이 반영 정책

### 노출 우선순위

1. 현재 주제
2. 미해결 질문
3. 확정 결정
4. 확정 액션 아이템
5. 활성 리스크
6. 라이브 컨텍스트

### 유지 시간 기준

- 현재 주제: 교체 전까지 유지
- 미해결 질문: 해결 전까지 유지
- 결정사항: 회의 종료 전까지 유지 가능
- 액션 아이템: 회의 종료 전까지 유지 가능
- 컨텍스트: 짧게 노출 후 교체

### 자동 접힘 기준

- 해결된 질문
- 중요도 낮은 컨텍스트
- 오래된 중복 리스크

---

## 9. 문서화 반영 정책

### 반영 우선순위

- `confirmed` 결정
- `confirmed` 액션 아이템
- `answered` 또는 `unresolved` 질문
- `active` 또는 `resolved` 리스크
- 주제 이력

### 반영 제외 또는 축소 대상

- 단순 라이브 컨텍스트
- 근거가 약한 후보 이벤트
- 중복되거나 충돌하는 후보 항목

### 문서화 시 정리 규칙

- 후보 이벤트는 그대로 내보내지 않는다.
- 사람이 읽기 쉬운 문장으로 정리한다.
- 액션 아이템은 가능한 한 표 또는 구조화된 목록으로 출력한다.

---

## 10. 예외 처리 정책

### 질문만 있고 답이 없는 경우

- 질문은 `unresolved`로 문서에 남긴다.

### 담당자 없는 액션 아이템

- 문서에는 작업 내용만 남기고 담당자는 비워 둔다.

### 상충되는 결정이 나온 경우

- 최신 확정 결정을 우선한다.
- 필요하면 이전 결정을 변경 이력으로 남긴다.

### STT 신뢰도가 낮은 경우

- 후보 이벤트로만 유지하거나 오버레이 노출을 낮춘다.

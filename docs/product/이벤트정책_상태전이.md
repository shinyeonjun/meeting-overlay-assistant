# 이벤트정책 / 상태전이

이 문서는 이벤트가 언제 만들어지고, 어떤 상태로 움직이며, UI와 리포트에서 어떻게 다르게 다뤄지는지 설명합니다.

현재 공식 기준 경로는 `server / client / shared / deploy`입니다.

## 1. 이벤트의 역할

현재 이벤트는 회의 중과 회의 후 사이를 잇는 작업 단위입니다.

이벤트는 아래 목적에 사용합니다.

- 질문 검토
- 결정 사항 정리
- 액션 아이템 추적
- 리스크 기록

## 2. 현재 이벤트 타입

현재 시스템은 아래 이벤트 타입을 다룹니다.

- `question`
- `decision`
- `action_item`
- `risk`

## 3. 실시간 경로와 최종 분석 경로의 차이

현재 live 경로에서는 `question`만 실시간으로 제안합니다.

이유는 다음과 같습니다.

- precision이 상대적으로 높다.
- 문장 패턴이 비교적 명확하다.
- UI를 단순하게 유지할 수 있다.

반면 아래 타입은 최종 분석에서 더 안정적으로 다룹니다.

- `decision`
- `action_item`
- `risk`

즉, 현재 정책은 `live question + final full analysis`입니다.

## 4. 이벤트 생성 경로

### live 생성

- 오디오 파이프라인이 질문 후보를 검출합니다.
- 세션에 연결된 이벤트로 저장합니다.
- 클라이언트는 질문 카드로 검토합니다.

### final 분석 생성

- 리포트 분석 단계에서 전체 transcript와 맥락을 봅니다.
- question / decision / action item / risk를 모두 분석에 사용합니다.

## 5. 현재 상태 모델

내부적으로는 상태가 저장되지만, UI는 이를 직접 설명하지 않습니다.

현재 핵심 상태는 아래와 같습니다.

- `open`
- `answered`
- `confirmed`
- `resolved`

## 6. 타입별 기본 상태 전이

### `question`

- 초기: `open`
- 사용자가 처리 완료하면: `answered`

### `decision`

- 초기: `open`
- 사용자가 처리 완료하면: `confirmed`

### `action_item`

- 초기: `open`
- 사용자가 처리 완료하면: `confirmed`

### `risk`

- 초기: `open`
- 사용자가 처리 완료하면: `resolved`

현재는 UI에서 복잡한 상태 머신을 노출하지 않고, 내부 정책으로만 관리합니다.

## 7. UI 행동 정책

현재 클라이언트에서 사용자가 보는 행동은 최소화합니다.

- `수정`
- `처리 완료`

이렇게 단순화한 이유는 아래와 같습니다.

- 회의 중에는 빠른 검토가 중요하다.
- 내부 상태 머신을 사용자에게 설명할 필요가 없다.
- 제품이 운영 콘솔처럼 느껴지지 않게 해야 한다.

## 8. 삭제보다 상태 전이를 우선하는 이유

현재 이벤트는 가능하면 삭제하지 않고 상태를 전이합니다.

이유는 다음과 같습니다.

- 리포트와 히스토리의 근거를 보존할 수 있다.
- 감사와 회귀 추적이 쉬워진다.
- “없앴다”보다 “처리했다”가 업무 흐름에 더 맞다.

## 9. 리포트 반영 정책

리포트 생성 시 이벤트는 상태와 타입에 따라 정리되어 반영됩니다.

현재 기준:

- 질문은 남은 질문이나 답변 완료 상태로 재해석될 수 있다.
- 결정 사항은 `confirmed` 여부를 기준으로 정리한다.
- 액션 아이템은 확인된 항목을 중심으로 정리한다.
- 리스크는 해결 여부를 함께 본다.

즉, 이벤트는 UI 카드 그대로 복사되는 것이 아니라 리포트 단계에서 다시 문서화됩니다.

## 10. 향후 확장 방향

현재 이후 확장 방향은 아래와 같습니다.

- event edit 근거 문장 표시 강화
- carry-over와 이벤트 연결
- 리포트 공유 이후 후속 조치 추적
- 참여자/contact 기준 액션 아이템 귀속

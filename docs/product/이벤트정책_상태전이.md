# 이벤트 정책 / 상태 전이

## 1. 목적

이 문서는 현재 프로젝트에서 이벤트가 어떤 의미를 가지는지, 어떤 상태를 갖는지, UI에서는 어떻게 단순화해서 보여주는지를 설명한다.

현재 핵심 원칙은 다음과 같다.

- 사용자에게는 단순한 액션만 노출한다.
- 내부적으로는 상태를 유지한다.
- live 인사이트는 질문만 다룬다.
- 최종 리포트는 더 풍부한 이벤트 타입을 사용한다.

## 2. 현재 이벤트 타입

최종적으로 지원하는 이벤트 타입은 다음 네 가지다.

- `question`
- `decision`
- `action_item`
- `risk`

### live 경로

현재 실시간 인사이트는 `question`만 노출한다.

### report 경로

최종 리포트 분석에서는 아래 네 가지를 모두 사용할 수 있다.

- question
- decision
- action_item
- risk

## 3. 사용자 액션과 내부 상태의 차이

### 사용자에게 보이는 액션

- `수정`
- `처리 완료`

### 내부적으로 유지하는 상태

- `open`
- `answered`
- `unresolved`
- `confirmed`
- `updated`
- `resolved`
- `closed`
- 기타 타입별 상태

즉 UI는 단순하지만, 서비스 계층은 상태 전이를 유지한다.

## 4. 처리 완료 매핑

현재 기본 UX에서 `처리 완료`는 타입별로 아래 상태에 대응한다.

- question -> `answered`
- decision -> `confirmed`
- action_item -> `confirmed`
- risk -> `resolved`

UI에서는 처리 완료 후 기본 보드에서 바로 숨긴다.  
단, DB에서는 삭제하지 않는다.

## 5. 타입별 상태 전이 개요

### question

대표 흐름:

- `open -> answered`
- `open -> unresolved`
- `open -> closed`

현재 기본 UI는 `처리 완료`로 `answered` 쪽에 가깝게 단순화한다.

### decision

대표 흐름:

- `candidate -> confirmed`
- `confirmed -> updated`
- `updated -> closed`

현재 live 경로에서는 거의 다루지 않고, 최종 분석/리포트 경로에서 더 중요하다.

### action_item

대표 흐름:

- `candidate -> confirmed`
- `confirmed -> updated`
- `updated -> closed`

### risk

대표 흐름:

- `open -> active`
- `active -> monitoring`
- `monitoring -> resolved`
- `resolved -> closed`

현재 기본 UI는 `처리 완료 -> resolved` 성격으로 단순화한다.

## 6. 수정 정책

현재 프론트에서는 이벤트 수정 시 제목 중심으로 다룬다.

이유:

- 오버레이 UI에서 과한 편집 기능은 복잡도를 크게 올린다.
- 현재 단계에선 빠른 검토와 보정이 더 중요하다.

향후 필요 시:
- 담당자
- 기한
- 본문
같은 필드를 확장할 수 있다.

## 7. 삭제 정책

현재 기본 UX에서는 삭제를 메인 액션으로 노출하지 않는다.

이유:

- 사용자는 “잘못 뽑힌 이벤트를 없애는 것”보다 “정리하고 넘기는 것”이 더 많다.
- 삭제는 이력과 근거를 날려버리므로 기본 플로우에 적합하지 않다.

API 차원에선 삭제가 남아 있어도, UX 차원에선 보조 기능으로 취급한다.

## 8. 리포트 반영 정책

리포트는 raw 이벤트를 그대로 보여주지 않고, 상태와 품질을 반영해 선별해야 한다.

권장 반영 방향:

- question: `open`, `answered`, `unresolved`
- decision: `confirmed`, `updated`
- action_item: `confirmed`, `updated`
- risk: `active`, `monitoring`, `resolved`

즉, 후보성 상태나 노이즈가 섞인 이벤트는 최종 문서에 그대로 넣지 않는다.

## 9. 현재 UX의 의도

현재 이벤트 UX는 상태 머신을 그대로 보여주는 것이 목표가 아니다.

목표:
- 빠르게 보고
- 필요한 질문만 수정하고
- 처리 완료로 넘기고
- 최종 리포트 생성으로 이어가기

즉 이벤트 탭은 “운영용 상태 콘솔”이 아니라 “빠른 검토 화면”이다.

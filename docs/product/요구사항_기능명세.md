# 요구사항 / 기능 명세

이 문서는 현재 CAPS가 해결하려는 문제, 제품 목표, 기능 범위, 그리고 이번 요구분석명세서 기준의 우선 구현 방향을 정리한다.

## 제품 정의

CAPS는 회의 중에는 `overlay`로 실시간 보조를 제공하고, 회의 후에는 `web workspace`에서 transcript, report, retrieval을 통해 정리와 후속 작업을 지원하는 로컬 중심 회의 보조 시스템이다.

## 문제 정의

1. 회의 중 핵심 발언, 결정, 액션 아이템을 놓치기 쉽다.
2. 회의 후 정리 문서를 만드는 데 시간이 오래 걸린다.
3. 이전 회의 맥락이 다음 회의 준비로 자연스럽게 이어지지 않는다.
4. 실시간 자막과 최종 노트에 요구되는 품질이 서로 다르다.

## 제품 목표

1. 회의 중 실시간 보조
2. 회의 후 정식 노트 / 리포트 생성
3. account / contact / thread 기반 맥락 추적
4. 과거 회의 retrieval과 다음 회의 준비
5. 사용자 기반 리포트 공유

## 핵심 사용자

### 1. 회의 진행자

- 회의를 시작하고 종료한다.
- 참가자와 맥락 정보를 입력한다.
- 회의 중 놓치면 안 되는 질문, 결정, 액션을 빠르게 확인한다.

### 2. 일반 참석자

- 실시간 자막과 이벤트를 보며 회의 흐름을 따라간다.
- 회의 후 노트와 리포트를 확인한다.

### 3. 회의 정리 담당자

- 회의 종료 후 transcript와 report를 확인한다.
- 필요한 경우 일부 내용을 보정하거나 공유한다.
- 다음 회의 준비를 위해 carry-over와 retrieval brief를 확인한다.

## 현재 구현 범위

### 인증

- workspace role 기반 권한 제어

### 세션

### history / retrieval

- timeline
- carry-over
- retrieval brief
- retrieval search

## 이번 차수의 집중 요구사항

이번 요구분석명세서에서는 전체 제품 범위를 유지하되, 특히 회의 노트 품질과 실시간 체감 품질을 아래 방향으로 개선하는 것을 우선 목표로 둔다.

### 1. live 경로와 final 경로를 분리한다

- live는 빠른 반응과 안정적인 partial 표시가 우선이다.
- final note는 느려도 더 정확한 transcript 생성이 우선이다.

### 2. 실시간 STT는 체감 품질을 우선한다

- live partial은 빠른 첫 응답과 흔들림 감소가 중요하다.
- 현재 비교 결과 기준으로 system audio의 live 경로는 sherpa 기반 hybrid 구성을 우선 검토한다.

### 3. final note는 정확도를 우선한다

- 회의 종료 후 생성하는 final note는 더 무거운 모델을 허용한다.
- 현재 검토 기준으로 final note STT는 Whisper `large-v3` 계열이 우선 후보이다.

### 4. transcript는 원문과 보정문을 분리한다

- raw transcript는 보존한다.
- corrected transcript는 별도로 생성한다.
- 노트와 리포트에는 corrected를 우선 반영할 수 있어야 한다.

### 5. LLM 보정은 보수적으로 제한한다

- 의미를 바꾸는 rewrite는 허용하지 않는다.
- 숫자, 날짜, 금액, 사람 이름, 회사 이름, 버전 정보는 보호 대상이다.
- transcript correction은 전체 발화 일괄 보정보다 의심 구간 중심으로 제한한다.

### 6. 용어 사전과 사용자 수정 흐름을 지원한다

- 제품명, 회사명, 영어 약어, 사람 이름을 보정 가능한 사전으로 관리한다.
- 사용자가 수정한 표현은 추후 재사용 가능한 구조를 고려한다.

## 기능 요구사항

### 세션 준비

- 사용자는 회의 시작 전에 account / contact / thread를 선택할 수 있어야 한다.
- 사용자는 참가자 정보를 입력할 수 있어야 한다.
- 사용자는 이전 회의의 carry-over / retrieval brief를 확인할 수 있어야 한다.

### 회의 진행

- 사용자는 overlay에서 live caption을 확인할 수 있어야 한다.
- 시스템은 질문 / 결정 / 액션 / 리스크를 실시간 이벤트로 추출할 수 있어야 한다.
- 사용자는 필요한 이벤트만 선택적으로 확인하거나 보정할 수 있어야 한다.

### 회의 종료 후 처리

- 시스템은 세션 종료 후 canonical transcript를 생성할 수 있어야 한다.
- 시스템은 transcript correction 결과를 별도 저장할 수 있어야 한다.
- 시스템은 markdown / pdf report를 생성할 수 있어야 한다.
- 시스템은 report generation 상태를 추적할 수 있어야 한다.

### 기록과 조회

- 사용자는 history timeline에서 과거 회의를 조회할 수 있어야 한다.
- 사용자는 retrieval search로 과거 회의 내용을 다시 찾을 수 있어야 한다.
- 사용자는 carry-over / retrieval brief를 통해 다음 회의 준비를 할 수 있어야 한다.

## 현재 제외 범위

- 별도 감사 로그 제품 기능
- OCR / screen context
- 별도 Vector SaaS

## 정책 메모

- 리포트 공유는 유지한다.
- 리포트 정본은 파일 artifact 기준이다.
- retrieval은 PostgreSQL + pgvector 기준이다.
- STT 품질 개선은 모델 교체만이 아니라 live / final 분리, 보수적 correction, 용어 사전, 사용자 수정 흐름까지 포함한 제품 문제로 본다.

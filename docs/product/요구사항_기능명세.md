# 요구사항 / 기능 명세

## 1. 제품 목표

이 프로젝트의 목표는 로컬 환경에서 회의 음성을 다루는 보조 도구를 제공하는 것이다.

핵심 가치는 두 가지다.

1. **회의 중 보조**
   - 실시간 자막
   - 질문 중심 인사이트
2. **회의 후 문서화**
   - 세션 녹음 기반 고정밀 STT
   - 수동 리포트 생성
   - Markdown / PDF 산출물

즉, 단순 STT 데모가 아니라 `실시간 보조 + 종료 후 문서화`를 함께 제공하는 것이 목표다.

## 2. 핵심 사용자 시나리오

### 시나리오 A: 오버레이 기반 회의 진행

1. 사용자가 앱을 실행한다.
2. readiness 상태를 확인한다.
3. 세션을 시작한다.
4. 회의 중 오버레이에서 자막을 본다.
5. 실시간 질문 이벤트를 확인한다.
6. 세션을 종료한다.
7. 필요 시 Markdown 또는 PDF 리포트를 생성한다.

### 시나리오 B: 회의 후 정리

1. 사용자가 이벤트 탭에서 질문을 검토한다.
2. 필요한 이벤트를 수정하거나 처리 완료한다.
3. 세션 탭에서 리포트 형식을 선택한다.
4. 리포트를 수동 생성한다.
5. 생성된 Markdown/PDF를 확인한다.

## 3. 현재 구현 기능

### 세션

- 세션 생성
- 세션 종료
- readiness 기반 시작 가능 상태 표시
- 실제 활성 입력 소스 기록
- overview 조회

### 실시간 자막

- `system_audio`, `mic`, `mic_and_audio` 지원
- partial / fast-final / final stability 반영
- 메인 live 자막과 누적 transcript 분리 표시

### 이벤트

- live 경로에서 질문 이벤트 생성
- 이벤트 목록 조회
- 이벤트 수정
- 처리 완료
- 상태 전이 API 유지

### 리포트

- 세션 종료 후 수동 생성
- Markdown 생성
- PDF 생성
- 리포트 버전 관리
- transcript / analysis artifact 저장
- 세션별 폴더 구조 정리

## 4. 현재 기능 범위 바깥

다음 항목은 코드 일부나 실험 흔적이 있어도, 현재 사용자 기능으로는 확정하지 않는다.

- 완전 자동 리포트 생성
- 실시간 질문 외 인사이트 노출
- 다자 화자 구분 완성
- 실시간 액션 아이템 자동 확정
- 다중 워크스페이스 / 협업 기능

## 5. 입력 소스별 기대 동작

### `system_audio`

- Tauri 브리지로 시스템 출력 오디오 캡처
- 회의, 유튜브, 발표 영상, 온라인 미팅 음성 입력에 사용
- 현재 기준 핵심 테스트 대상

### `mic`

- 마이크 음성 입력
- 텍스트 WebSocket 또는 브라우저 음성 인식 보조 경로와 함께 사용 가능

### `mic_and_audio`

- 마이크와 시스템 오디오를 함께 다루는 세션 모드
- 실제 활성 입력 소스는 세션에 기록

## 6. 리포트 기능 명세

### 리포트 생성 정책

- 자동 생성하지 않는다.
- 세션 종료 후 사용자가 수동으로 생성한다.
- 형식은 `markdown`, `pdf` 중 선택한다.

### 리포트 생성 입력

리포트 생성 시 백엔드는 아래 우선순위로 입력을 찾는다.

1. 명시적으로 전달된 `audio_path`
2. 세션 녹음 파일 자동 탐색
3. 녹음 파일이 없으면 `live_fallback`

### 리포트 출력

세션 폴더 아래에 저장한다.

```text
backend/data/reports/{session_id}/
  markdown.v1.md
  pdf.v1.pdf
  artifacts/
    *.transcript.md
    *.analysis.json
```

### 리포트 품질 목표

- raw dump가 아니라 사용자용 구조화 문서
- 섹션별 가독성 확보
- 참고 전사는 시간대 기준으로 읽기 쉬운 형식 유지

## 7. 이벤트 기능 명세

### 현재 사용자 기준 액션

- `수정`
- `처리 완료`

이유:
- 상태 머신을 그대로 노출하면 UX가 나빠짐
- 내부적으로는 상태 전이를 유지하되, 사용자 노출은 최소화함

### 처리 완료 의미

- question -> answered
- decision -> confirmed
- action_item -> confirmed
- risk -> resolved

UI에서는 처리 완료 후 기본 보드에서 숨긴다.

## 8. 품질 관련 남은 과제

현재 구현은 기능적으로는 닫혔지만, 품질 측면에서는 세 가지가 남아 있다.

1. STT 품질
2. 이벤트 추출 precision
3. 리포트 분석 품질

즉 현재 단계는 “새 기능 추가”보다 “결과 품질 개선”이 핵심이다.

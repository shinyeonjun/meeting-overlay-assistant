# 요구사항 분석 및 기능 명세서

> 프로젝트명: 로컬 AI 회의 오버레이 어시스턴트  
> 문서 목적: 제품이 해결하려는 문제, 사용자 요구, 기능 범위, 우선순위, 수용 기준을 정의한다.

---

## 1. 문서 개요

이 문서는 아래 질문에 답하기 위해 작성한다.

- 이 제품은 누구의 어떤 문제를 해결하는가
- 회의 중 기능과 회의 후 기능은 각각 어디까지 해야 하는가
- MVP에서 반드시 들어가야 하는 기능은 무엇인가
- 구현 과정에서 어떤 품질 기준을 만족해야 하는가

이 문서는 기획서와 설계서 사이의 기준 문서다.

- `계획.md`는 제품 방향과 개발 계획을 설명한다.
- `설계.md`는 시스템 구조와 처리 흐름을 설명한다.
- 이 문서는 실제 구현 대상과 우선순위를 결정하는 기준이 된다.

---

## 2. 제품 배경

회의에서는 아래 문제가 반복된다.

- 중요한 질문이 나왔는데 회의 중간에 묻힌다.
- 결정된 내용이 명확히 남지 않는다.
- 누가 무엇을 언제까지 하기로 했는지 회의 직후부터 흐려진다.
- 회의가 끝난 뒤 회의록을 다시 정리하는 데 시간이 많이 든다.
- 회의 중 화면과 대화가 같이 흘러가는데, 사람이 핵심 맥락을 계속 붙들고 있기 어렵다.

기존 도구의 한계는 다음과 같다.

- 자막 도구는 발화를 그대로 보여주지만 중요한 정보만 구조화하지는 못한다.
- 요약 도구는 회의 후 결과는 만들 수 있어도 회의 중 진행 보조는 약하다.
- 일반 노트 도구는 사용자가 직접 기록해야 한다.

따라서 본 제품은 다음 두 문제를 동시에 해결해야 한다.

1. `회의 중 문제`
- 지금 당장 중요한 질문, 결정, 액션 아이템을 놓치지 않게 도와야 한다.

2. `회의 후 문제`
- 회의가 끝난 뒤 공유 가능한 수준의 문서를 빠르게 만들어야 한다.

```text
회의 중에는 중요한 것을 실시간에 가깝게 구조화해 보여주고,
회의 후에는 실행 가능한 문서로 정리해주는 로컬 AI 회의 어시스턴트
```

## 4. 사용자 및 이해관계자

### 4.1 주요 사용자

1. 개발팀 / 캡스톤 팀
- 회의 중 결정사항과 TODO를 바로 확인하고 싶은 사용자

2. 발표 준비자
- 발표 리허설 또는 논의 중 나온 핵심 질문과 수정 포인트를 놓치고 싶지 않은 사용자

3. 회의 기록 담당자
- 회의가 끝난 뒤 회의록을 처음부터 다시 쓰고 싶지 않은 사용자

### 4.2 이해관계자

- 발표자 또는 회의 리더
- 회의 참가자
- 프로젝트 지도교수 또는 심사자
- 향후 유지보수 / 구현 담당자

## 5. 사용 시나리오

### 시나리오 A. 주간 개발 회의

- 팀이 버그 원인과 배포 범위를 논의한다.
- 제품은 질문, 결정사항, 액션 아이템을 카드로 띄운다.
- 회의 종료 후 결과 문서를 자동으로 생성한다.

### 시나리오 B. 캡스톤 진행 회의

- 역할 분담과 일정 논의를 한다.
- 제품은 담당자와 기한이 포함된 액션 아이템을 우선적으로 잡아낸다.
- 종료 후 문서에서 주제 흐름과 미해결 질문을 정리한다.

### 시나리오 C. 발표 리허설

- 발표 중 나온 피드백과 수정 요청을 기록한다.
- 제품은 리스크와 후속 조치를 문서화한다.

- 화자 분리 정확도 고도화
- 감정 분석
- 제스처 인식
- 클라우드 기반 협업 문서 동기화
- 대규모 다중 조직용 SaaS 기능

## 7. 핵심 가치 제안

### 회의 중 가치

- 회의에서 놓치기 쉬운 핵심 포인트를 빠르게 드러낸다.
- 사람이 메모하느라 회의 흐름을 놓치는 문제를 줄인다.
- 미해결 질문과 확정된 결정을 시각적으로 구분해 보여준다.

### 회의 후 가치

- 회의 직후 공유 가능한 문서를 빠르게 확보한다.
- 질문, 결정, 액션 아이템이 섞이지 않고 구조화된다.
- 회의 내용의 후속 실행력을 높인다.

## 8. 기능 구조

본 제품의 기능은 두 개의 상위 기능군으로 구성한다.

1. `회의 중 기능`
- 실시간에 가까운 이벤트 추출
- 오버레이 표시
- 상태 갱신

2. `회의 후 기능`
- 누적 데이터 재정리
- 문서 생성
- 결과 조회 및 내보내기

---

## 9. 기능 요구사항

### 9.1 세션 관리

#### FR-001 세션 시작

- 사용자는 회의 제목, 모드, 오디오 입력 소스를 지정해 세션을 시작할 수 있어야 한다.
- 시스템은 세션 시작 시각과 상태를 저장해야 한다.

수용 기준:
- 세션 생성 요청 후 고유 ID가 반환된다.
- 세션 상태는 `running`으로 저장된다.

#### FR-002 세션 종료

- 사용자는 회의 종료 시 세션을 종료할 수 있어야 한다.
- 시스템은 종료 시각을 저장하고 최종 문서 생성 플로우를 시작해야 한다.

수용 기준:
- 세션 종료 요청 후 상태가 `ended`로 바뀐다.
- 문서 생성 작업이 시작된다.

---

### 9.2 오디오 및 화면 입력

#### FR-003 오디오 입력 수집

- 시스템은 시스템 오디오 또는 마이크 입력을 받을 수 있어야 한다.
- 입력은 발화 단위 처리를 위해 chunk 형태로 수집되어야 한다.
- 입력 소스별 기본 경로는 아래와 같이 분리한다.
  - `system_audio`: backend WebSocket 기반 로컬 하이브리드 STT
  - `mic`: Web Speech API 기반 STT(기본), 실패 시 backend WebSocket STT fallback

수용 기준:
- 설정된 입력 소스에서 오디오 수집이 시작된다.
- 수집 데이터가 후속 STT 단계로 전달된다.

#### FR-004 발화 구간 분리

- 시스템은 연속된 오디오를 발화 단위로 분리해야 한다.

수용 기준:
- 발화 시작 / 종료 구간이 시간 정보와 함께 기록된다.

#### FR-005 화면 맥락 수집

- 시스템은 일정 주기 또는 이벤트 기반으로 화면을 캡처할 수 있어야 한다.
- 캡처 결과에서 OCR 텍스트와 제목 힌트를 추출할 수 있어야 한다.

수용 기준:
- 화면 캡처 시각이 기록된다.
- OCR 텍스트 또는 화면 제목 힌트가 저장된다.

---

### 9.3 회의 중 기능

#### FR-006 현재 주제 추출

- 시스템은 최근 발화와 화면 맥락을 바탕으로 현재 주제를 추정해야 한다.
- 현재 주제는 한 시점에 1개만 활성 상태로 유지해야 한다.

수용 기준:
- 새로운 주제가 감지되면 기존 주제는 `replaced` 처리된다.
- 프론트에는 현재 주제 1개만 노출된다.

#### FR-007 질문 감지

- 시스템은 질문성 발화를 감지해 질문 이벤트를 생성해야 한다.
- 질문이 해결되면 기존 질문 상태를 업데이트해야 한다.

수용 기준:
- 질문 감지 시 `question_detected` 이벤트가 생성된다.
- 답변 연결 시 `question_resolved` 이벤트가 생성되거나 상태가 변경된다.

#### FR-008 결정사항 감지

- 시스템은 회의 중 확정 표현을 감지해 결정사항을 생성해야 한다.

수용 기준:
- 결정사항은 `decision` 이벤트로 저장된다.
- 프론트에서 최근 결정사항으로 노출된다.

#### FR-009 액션 아이템 감지

- 시스템은 담당자, 작업 내용, 기한이 포함된 후속 작업을 추출해야 한다.
- 정보가 불완전한 경우 후보 상태로 저장할 수 있어야 한다.

수용 기준:
- 액션 아이템은 `candidate` 또는 `confirmed` 상태를 가진다.
- 담당자 또는 기한이 확인되면 기존 이벤트가 갱신된다.

#### FR-010 리스크 감지

- 시스템은 장애, 우려사항, 미확인 리스크를 감지해야 한다.

수용 기준:
- 리스크는 `risk` 이벤트로 저장된다.
- 중요도에 따라 오버레이 표시 여부를 결정할 수 있다.

#### FR-011 라이브 맥락 갱신

- 시스템은 최근 발화와 화면 제목 / 키워드를 프론트에 전달해야 한다.

수용 기준:
- 라이브 맥락은 짧은 텍스트 단위로 갱신된다.
- 낮은 중요도의 맥락은 접히거나 생략될 수 있다.

#### FR-012 오버레이 상태 유지

- 시스템은 회의 중 표시 중인 카드 상태를 유지하고 갱신해야 한다.

수용 기준:
- 미해결 질문은 쉽게 사라지지 않는다.
- 결정사항과 액션 아이템은 회의 중 계속 유지될 수 있다.

---

### 9.4 회의 후 기능

#### FR-013 회의 전체 재정리

- 세션 종료 후 시스템은 누적된 발화, 화면 맥락, 이벤트를 다시 읽어 정리해야 한다.

수용 기준:
- 중복 이벤트가 정리된다.
- 최종 문서에 들어갈 구조가 재구성된다.

#### FR-014 전체 요약 생성

- 시스템은 회의의 전체 요약을 생성해야 한다.

수용 기준:
- 문서 첫 부분에 2~5개 수준의 핵심 요약 항목이 포함된다.

#### FR-015 주제 흐름 생성

- 시스템은 회의 전체 주제 흐름을 순서대로 정리해야 한다.

수용 기준:
- 문서에 주제 흐름 섹션이 포함된다.

#### FR-016 질문 / 답변 정리

- 시스템은 질문과 연결된 답변을 문서 구조에 맞게 정리해야 한다.
- 해결되지 않은 질문은 별도로 표시해야 한다.

수용 기준:
- 해결된 질문과 미해결 질문이 구분된다.

#### FR-017 결정사항 정리

- 시스템은 최종 결정사항을 중복 없이 정리해야 한다.

수용 기준:
- 문서에 결정사항 목록이 포함된다.

#### FR-018 액션 아이템 정리

- 시스템은 담당자, 작업 내용, 기한을 포함한 액션 아이템 목록을 정리해야 한다.

수용 기준:
- 액션 아이템은 최소한 담당자 또는 작업 내용이 포함되어야 한다.
- 기한이 없으면 비워 둘 수 있다.

#### FR-019 리스크 및 후속 확인사항 정리

- 시스템은 리스크와 후속 확인이 필요한 항목을 문서로 정리해야 한다.

수용 기준:
- 문서에 리스크 섹션이 포함된다.

#### FR-020 문서 생성 및 조회

- 시스템은 Markdown, HTML, PDF 형태의 결과 문서를 생성할 수 있어야 한다.
- 사용자는 생성된 문서를 조회할 수 있어야 한다.

수용 기준:
- 세션별 리포트 목록을 조회할 수 있다.
- 최소 Markdown 결과물은 항상 생성 가능해야 한다.

### 9.5 관리 및 조회 기능

#### FR-021 세션 전체보기 조회

- 사용자는 회의 중 또는 회의 후에 세션 overview를 볼 수 있어야 한다.

수용 기준:
- 현재 주제, 질문, 결정, 액션, 리스크 상태를 조회할 수 있다.

#### FR-022 최종 문서 생성 상태 조회

- 사용자는 세션 종료 후 문서 생성 상태를 확인할 수 있어야 한다.

수용 기준:
- `pending`, `processing`, `completed`, `failed` 중 하나의 상태를 제공한다.

## 10. 비기능 요구사항

### 10.1 성능

#### NFR-001 실시간 반응성

- 회의 중 기능은 사용자가 실시간에 가깝다고 느낄 수준으로 동작해야 한다.

목표 기준:
- 주요 이벤트는 발화 후 수 초 내 반영을 목표로 한다.

#### NFR-002 문서 생성 시간

- 회의 종료 후 최종 문서는 과도한 대기 없이 생성되어야 한다.

목표 기준:
- 짧은 회의 기준 수십 초 내 1차 결과 확인 가능

### 10.2 신뢰성

#### NFR-003 상태 일관성

- 회의 중 카드 상태와 회의 후 문서 내용은 같은 사실 구조를 공유해야 한다.

#### NFR-004 장애 대응

- 일부 추론 경로가 실패해도 fallback 또는 축소 기능으로 동작을 유지해야 한다.
- `mic` 경로는 Web Speech API가 기본이며, 장애/미지원 시 backend STT fallback 전환 정책을 적용한다.

### 10.3 사용성

#### NFR-005 오버레이 최소 방해

- 오버레이는 회의 화면을 과도하게 가리면 안 된다.
- 항상 중요한 정보만 보여줘야 한다.

#### NFR-006 문서 가독성

- 최종 문서는 바로 공유 가능한 수준의 구조와 가독성을 가져야 한다.

### 10.4 보안 및 개인정보

#### NFR-007 로컬 저장 우선

- 오디오, 발화, 화면 맥락, 문서는 기본적으로 로컬 환경에 저장되는 것을 우선한다.

#### NFR-008 개인정보 및 민감정보 주의

- 한국 맥락에서 음성, 화면, 회의 내용은 개인정보 및 업무상 민감정보를 포함할 수 있다.
- 사용자에게 수집 사실과 저장 범위를 명확히 안내할 수 있어야 한다.

#### NFR-009 보존 기간 및 삭제 가능성

- 세션 데이터와 문서는 삭제 가능해야 한다.
- 추후 보존 기간 정책을 붙일 수 있어야 한다.

## 12. MVP 정의

MVP는 다음 조건을 만족하면 된다.

- 회의를 시작할 수 있다.
- 오디오 입력을 받아 핵심 이벤트를 추출할 수 있다.
- 회의 중 현재 주제, 질문, 결정, 액션 아이템을 오버레이로 표시할 수 있다.
- 회의 종료 후 Markdown 문서를 생성할 수 있다.
- 문서에 질문, 결정, 액션 아이템이 구조화되어 포함된다.

즉, MVP의 본질은 다음과 같다.

```text
회의 중에는 중요한 정보를 놓치지 않게 하고,
회의 후에는 공유 가능한 문서를 만드는 것
```

## 13. 수용 기준 요약

### 회의 중 기능 수용 기준

- 사용자는 회의 도중 현재 주제를 볼 수 있다.
- 사용자는 미해결 질문을 확인할 수 있다.
- 사용자는 최근 결정사항과 액션 아이템을 확인할 수 있다.
- 오버레이는 회의 흐름을 과도하게 방해하지 않는다.

### 회의 후 기능 수용 기준

- 세션 종료 후 문서 생성이 시작된다.
- 문서에는 전체 요약, 질문 / 답변, 결정사항, 액션 아이템이 포함된다.
- 최소 하나 이상의 문서 포맷이 생성된다.

### 발표 관점 수용 기준

- 짧은 데모 회의에서 카드 생성 흐름이 눈에 보인다.
- 회의 종료 후 결과 문서가 바로 제시된다.
- 회의 중 결과와 회의 후 문서가 연결된다는 점이 설명 가능하다.

## 14. 리스크 및 쟁점

- STT 품질이 낮으면 질문 / 액션 감지 정확도가 흔들릴 수 있다.
- OCR 품질이 낮으면 현재 주제 추정이 흔들릴 수 있다.
- 실시간성과 정확도 사이의 균형을 잘못 잡으면 둘 다 어설퍼질 수 있다.
- 개인정보, 회의 녹음 동의, 민감정보 저장 이슈를 고려해야 한다.

# 운영정책 / 비기능

현재 CAPS의 운영 비기능 기준을 정리한다.

## 목표

- 실시간성
- 정확도
- 운영 추적 가능성
- 확장 가능성
- 개인정보 보호

## 저장소 정책

- 정본 DB: PostgreSQL
- retrieval 계층: pgvector
- 비동기 queue: Redis
- artifact: raw audio, transcript, markdown, pdf
- 테스트 격리: PostgreSQL 전용 테스트 DB

## live / final 분리 정책

### live

- 목표는 빠른 첫 반응과 안정적인 partial 표시다.
- partial latency, flicker, final latency를 핵심 지표로 본다.
- 실시간 경로는 마이크와 system audio를 동일하게 취급하지 않고 source별로 튜닝한다.

### final note

- 목표는 더 정확한 transcript와 신뢰 가능한 report다.
- WER, CER, 고유명사 / 숫자 보존률, human review를 핵심 지표로 본다.
- live보다 무거운 모델과 후처리를 허용한다.

## transcript 정책

- raw transcript는 보존한다.
- corrected transcript는 별도 저장한다.
- corrected는 노트 / 리포트 반영용 결과물로 취급한다.
- correction은 의미를 바꾸는 rewrite가 아니라 보수적 보정으로 제한한다.

## 리포트 정책

- 세션 종료와 리포트 생성은 분리한다.
- 리포트 생성은 `report_generation_job` 상태로 추적한다.
- worker가 transcript, markdown, pdf, indexing을 후처리한다.

## STT 개선 정책

- live와 final에 동일한 품질 기준을 적용하지 않는다.
- live는 partial 안정성과 반응성을 우선한다.
- final note는 더 정확한 STT 모델을 우선 검토한다.
- correction은 전 발화 일괄 적용보다 의심 구간 중심 적용을 우선 검토한다.
- 용어 사전, 사용자 수정 반영, correction 로그 축적을 중기 개선 축으로 둔다.

## 개인정보 / 민감정보

- 회의 음성, 발화 텍스트, 연락처 정보는 민감할 수 있다.
- 개인정보보호법 기준으로 보관 기간과 접근 권한을 분리해 관리해야 한다.
- raw audio는 목적 달성 후 삭제 가능한 구조를 우선한다.
- transcript와 report는 업무 산출물로 취급한다.

## 후속 확장 후보

- Redis lock / cache 고도화
- dedicated worker 확장
- 백업 / 복구 자동화
- 사용자 수정 기반 용어 사전 축적
- source별 STT 정책 고도화

# 운영 정책 및 비기능 설계

> 문서 목적: 성능, 신뢰성, 개인정보, 로깅, 장애 대응, 품질 기준 등 구현 후 운영에 직접 영향을 주는 정책을 정의한다.

---

## 1. 문서 역할

이 문서는 기능 명세서의 비기능 요구를 실제 운영 기준으로 구체화한다.

다루는 영역:

- 성능 목표
- 신뢰성 및 fallback
- 개인정보 및 민감정보 처리
- 데이터 보존 및 삭제
- 로깅과 모니터링
- 테스트 및 품질 기준

---

## 2. 운영 원칙

### 원칙 1. 로컬 우선

- 핵심 데이터는 기본적으로 로컬에서 처리하고 저장한다.

### 원칙 2. 축소 동작 가능

- 모든 기능이 완벽하게 동작하지 않아도 핵심 기능은 남아야 한다.

### 원칙 3. 설명 가능성 확보

- 왜 특정 이벤트가 떴고 왜 특정 문서가 만들어졌는지 추적 가능해야 한다.

### 원칙 4. 개인정보 최소화

- 필요한 데이터만 수집하고, 과도한 보존을 피한다.

---

## 3. 성능 기준

### 3.1 회의 중 성능 목표

- 사용자가 실시간에 가깝다고 느끼는 반응 속도를 목표로 한다.
- 카드 생성은 발화 직후 수 초 이내 반영을 목표로 한다.

핵심 지표:

- 오디오 입력 지연
- VAD 처리 지연
- STT 처리 지연
- 이벤트 추출 지연
- UI 반영 지연

### 3.2 회의 후 성능 목표

- 짧은 회의 기준으로 문서 생성이 과도하게 늦지 않아야 한다.
- 최소 1차 Markdown 결과는 빠르게 확인 가능해야 한다.

핵심 지표:

- 세션 종료 후 문서 생성 시작 시간
- 문서 생성 완료 시간
- 포맷별 생성 시간

---

## 4. 장애 대응 및 fallback 정책

### 4.1 STT 실패

대응:
- 해당 구간을 낮은 신뢰도 utterance로 저장하거나 건너뛴다.
- UI는 입력 중단 또는 품질 저하 상태를 표시할 수 있다.

### 4.2 OCR 실패

대응:
- OCR은 보조 입력이므로, 실패해도 회의 중 기능은 계속 동작해야 한다.
- 현재 주제 추정에서 OCR 가중치를 낮춘다.

### 4.3 NPU 경로 실패

대응:
- CPU fallback으로 전환한다.
- 성능 저하 상태를 내부 로그에 남긴다.

### 4.4 이벤트 추출 실패

대응:
- 원 발화 저장은 유지한다.
- 이벤트 추출 재시도 또는 회의 후 문서화 단계에서 보강한다.

### 4.5 문서 생성 실패

대응:
- 최소 Markdown 재시도
- 실패 사유 로그 저장
- 사용자에게 `failed` 상태 표시

### 4.6 mic 경로 장애

대응:
- 현재 mic 기본 경로(Web Speech API) 장애 시 backend STT fallback으로 전환한다.
- Web Speech API 사용 시 브라우저 권한/지원 여부를 먼저 검사한다.
- Web Speech API 실패 시 `mic backend fallback`으로 전환한다.
- 전환 사실과 원인을 세션 로그에 남긴다.

---

## 5. 개인정보 및 민감정보 정책

### 5.1 기본 인식

이 시스템은 아래 민감 데이터를 다룰 수 있다.

- 음성 데이터
- 발화 텍스트
- 화면 캡처 이미지
- 화면 OCR 텍스트
- 업무상 비밀 또는 개인식별정보가 포함된 회의 내용

### 5.2 한국 맥락 고려사항

- 회의 녹음 또는 화면 캡처는 조직 내 정책과 참가자 동의 이슈가 발생할 수 있다.
- 개인정보보호법, 사내 보안정책, 연구실 / 회사 정책을 고려해야 한다.
- `Web Speech API`를 사용할 경우 브라우저/플랫폼 정책에 따라 음성 데이터가 외부 처리될 수 있으므로 고지와 동의 절차를 둔다.

### 5.3 수집 최소화 정책

- 기본적으로 기능에 필요한 최소 단위만 저장한다.
- 화면 전체 이미지보다 OCR 결과와 제목 힌트를 우선 활용할 수 있도록 설계한다.
- 라이브 컨텍스트는 장기 저장보다 보조 노출 중심으로 설계한다.

### 5.4 사용자 안내 정책

최소한 아래 사실은 사용자에게 안내 가능해야 한다.

- 오디오가 수집 중인지
- `mic`가 backend STT 경로인지, Web Speech API 경로인지
- 화면 맥락이 수집 중인지
- 데이터가 로컬에 저장되는지
- 세션 종료 후 문서가 생성되는지

### 5.5 마스킹 및 확장 정책

추후 확장 시 고려:

- 이메일, 전화번호, 주민등록번호 패턴 마스킹
- 민감 키워드 포함 세션 경고
- 특정 리포트 내보내기 제한

## 6. 데이터 보존 및 삭제 정책

### 6.1 저장 대상

- sessions
- utterances
- screen_contexts
- overlay_events
- reports

### 6.2 보존 원칙

- MVP에서는 수동 삭제 가능 구조를 우선한다.
- 추후 자동 삭제 정책을 붙일 수 있게 설계한다.

### 6.3 삭제 정책

- 세션 삭제 시 관련 발화, 화면 맥락, 이벤트, 문서도 같이 정리할 수 있어야 한다.
- 문서만 개별 삭제 가능한 구조도 고려한다.

---

## 7. 로깅 정책

### 필수 로그

- 세션 시작 / 종료 로그
- 오디오 입력 시작 / 중단 로그
- 이벤트 생성 / 업데이트 로그
- 상태 전이 로그
- 문서 생성 시작 / 완료 / 실패 로그
- fallback 전환 로그

### 로그 원칙

- 디버깅 가능한 정보는 남기되 민감 원문을 무분별하게 남기지 않는다.
- 운영 로그와 분석 로그의 목적을 분리한다.

### 추후 확장

- 구조화 로그(JSON)
- 세션별 trace id
- 이벤트별 trace id

## 8. 모니터링 기준

### 회의 중

- 현재 입력 상태
- 최근 이벤트 생성 시각
- 오버레이 갱신 정상 여부

### 회의 후

- 문서 생성 진행 상태
- 실패 횟수
- 포맷별 생성 성공 여부

---

## 9. 품질 보증 기준

### 테스트 우선순위

1. 세션 시작 / 종료
2. 이벤트 생성 / 상태 전이
3. 오버레이 상태 집계
4. 문서 생성
5. fallback 동작

### 최소 테스트 시나리오

- 질문만 있는 회의
- 질문 + 답변이 있는 회의
- 결정과 액션이 동시에 나오는 회의
- OCR 없이 동작하는 회의
- NPU 없이 CPU fallback으로 동작하는 회의

### 수동 검증 포인트

- 카드가 과도하게 많이 뜨지 않는지
- 질문이 너무 빨리 사라지지 않는지
- 최종 문서에 중복 항목이 과하지 않은지

---

## 10. 발표 / 데모 운영 정책

### 데모 원칙

- 짧고 명확한 회의 시나리오를 사용한다.
- 질문, 결정, 액션 아이템이 모두 나오게 구성한다.
- 회의 종료 후 문서 결과를 바로 보여준다.

### 데모 안정화 팁

- 입력 소스 환경을 고정한다.
- 테스트용 시나리오를 미리 준비한다.
- CPU fallback 경로도 같이 점검한다.

---

## 11. 오픈 이슈

- 화면 캡처 이미지를 기본 저장할지, 옵션 저장으로 둘지
- 세션 단위 삭제 API를 MVP에 포함할지
- 개인정보 마스킹을 MVP에 넣을지 후순위로 둘지
- 문서 생성 실패 시 사용자 재시도 흐름을 어떻게 둘지

# 운영정책 / 비기능

## 1. 기본 운영 원칙

이 프로젝트는 로컬 데스크톱 환경에서 동작하는 회의 보조 MVP다.  
현재 비기능 정책은 “빠른 실험”보다 “운영 흐름의 일관성”을 우선한다.

핵심 원칙:

- live는 빠름 우선
- report는 정확함 우선
- UI는 단순화
- 내부 상태와 산출물은 보존

## 2. 리포트 생성 정책

### 현재 정책

- 세션 종료 시 자동 생성하지 않는다.
- 사용자가 명시적으로 생성한다.
- 형식은 Markdown 또는 PDF를 선택한다.

### 이유

- 현재 품질 수준에서 자동 생성은 과도하게 공격적이다.
- 사용자가 이벤트를 한 번 검토한 뒤 생성하는 편이 더 안전하다.
- 종료와 생성 실패를 분리하면 운영/디버깅이 쉬워진다.

## 3. 녹음 파일 정책

리포트는 세션 녹음 파일을 사용해 고정밀 STT를 수행한다.

### 저장 위치

- `backend/data/recordings/`

### 원칙

- Git 추적 대상에서 제외
- 사용자가 직접 경로를 입력하지 않음
- 세션 기준 자동 탐색

### 개인정보 관점

한국 맥락에서는 음성 파일 자체가 민감할 수 있으므로, 기본값은 장기 보관보다 처리용 임시 산출물에 가깝게 다루는 것이 맞다.

## 4. 리포트 저장 정책

### 저장 위치

```text
backend/data/reports/{session_id}/
```

### 최종 산출물

- `markdown.v{n}.md`
- `pdf.v{n}.pdf`

### 중간 산출물

- `artifacts/*.transcript.md`
- `artifacts/*.analysis.json`

### 원칙

- 세션 단위로 정리
- 버전 유지
- 최종본과 artifact 분리

## 5. 실시간 인사이트 정책

현재 live 경로는 질문만 노출한다.

### 이유

- question은 상대적으로 precision이 높다.
- decision/action_item/risk는 실시간 경로에서 오탐이 많았다.
- live 화면은 복잡성보다 안정성이 중요하다.

최종 문서 분석에서는 여전히 question/decision/action_item/risk를 모두 사용한다.

## 6. 상태 노출 정책

내부적으로는 이벤트 상태를 유지한다.

예:
- open
- answered
- confirmed
- resolved

하지만 UI에는 raw 상태명을 최소한으로만 드러낸다.

### 이유

- 사용자는 상태 머신을 조작하려는 것이 아니라 회의를 정리하려는 것이다.
- 상태 전이는 내부 규칙 검증과 리포트 반영에만 필요하다.

## 7. readiness 정책

세션 시작 전에는 `runtime/readiness`를 통해 준비 상태를 확인한다.

### 의도

- 프론트 브리지 준비 여부
- 백엔드 준비 여부
- STT preload 여부
를 분리해서 표시

### 운영 정책

- 시작 전 polling
- 세션 시작 후 polling 중단
- 세션 종료 후 polling 재개

## 8. STT 운영 정책

### 현재 전략

- 실시간: Sherpa 기반 partial / fast-final 중심
- 최종 기록: Faster-Whisper 기반 archive-final 및 리포트 STT

### live final 정책

- 늦은 final은 live 화면을 뒤집지 않는다.
- 저장/리포트 경로에만 반영한다.

### 이유

- 늦은 final이 실시간 화면을 자주 갈아치우면 사용자 체감이 나빠진다.

## 9. 품질 관리 대상

현재 운영상 핵심 품질 축은 세 가지다.

1. STT 품질
2. 이벤트 추출 precision
3. 리포트 분석 품질

이 세 축이 우선이며, 기능 추가는 그 다음이다.

## 10. 개발/운영 분리 정책

개발용 입력 경로, 샘플 fixture, benchmark 문서는 제품 경로와 분리한다.

예:
- `/api/v1/ws/text/{session_id}`: 개발/보조 입력
- `tests/video/benchmark*.json`: 연구 근거
- `docs/research/`: 설계 근거

즉 운영 문서의 단일 기준은 `architecture/`, `product/`이고, `research/`, `internal/`은 참고 자료다.

# 오버레이 UI/UX 명세

## 1. 화면 구조

현재 오버레이 앱은 크게 두 탭으로 구성된다.

- `세션`
- `이벤트`

과거에 있던 별도 `리포트` 탭은 제거했고, 리포트 생성 흐름은 세션 탭으로 통합했다.

## 2. 세션 탭

세션 탭은 사용자의 메인 작업 화면이다.

### 주요 역할

- readiness 확인
- 세션 시작
- 세션 종료
- live 자막 확인
- 현재 주제 확인
- 리포트 수동 생성

### 표시 항목

- readiness 상태
- 세션 상태
- 입력 소스
- 현재 주제
- live 자막 메인 뷰
- 누적 transcript
- 리포트 형식 선택
- 리포트 생성 버튼
- 리포트 생성 상태 / 최신 산출물 정보

### 세션 버튼 정책

- 시작 전에는 readiness가 충족될 때만 세션 시작을 허용한다.
- 종료는 `세션 종료`만 수행한다.
- 리포트 생성은 별도 버튼으로 수동 수행한다.

의도:
- 종료와 문서 생성을 분리해서 실패 원인을 명확히 한다.
- 사용자가 이벤트를 검토한 뒤 리포트를 생성할 수 있게 한다.

## 3. 메인 live 자막 영역

메인 live 자막은 “가장 최근의 현재 발화”를 보여준다.

### 동작 원칙

- 새 세그먼트의 불안정한 `low partial`이 들어와도, 이전의 더 안정된 문장을 바로 지우지 않는다.
- `medium` 또는 `final` 수준이 되면 메인 문장을 교체한다.
- 즉, 메인 영역은 “현재 듣고 있는 핵심 문장” 역할을 한다.

### 안정성 표현

- `low`: 가장 불안정한 partial
- `medium`: fast-final 성격의 안정된 중간 단계
- `final`: archive-final 또는 확정 텍스트

UI는 색상과 강조 정도로 이를 구분한다.

## 4. 누적 transcript 영역

메인 자막 아래 영역은 확정되거나 충분히 안정된 발화를 누적해서 보여준다.

의도:
- 메인 영역은 현재 발화 추적
- 누적 영역은 직전 맥락 회상

이 둘을 분리해야 실시간 자막이 “떴다 사라지는 문제”를 줄일 수 있다.

## 5. 이벤트 탭

이벤트 탭은 실시간 질문 이벤트를 검토하는 화면이다.

### 현재 UX 원칙

- 카드별 액션은 최소화한다.
- 상태명(`confirmed`, `resolved` 등)은 직접 노출하지 않는다.
- 사용자에게는 `수정`, `처리 완료` 정도만 보여준다.

### 카드 구성

- 이벤트 타입 라벨
- 제목
- 필요 시 근거 문장
- `수정`
- `처리 완료`

### 수정

- 오버레이 내부의 단일 입력 필드로 제목만 수정한다.
- 브라우저 기본 `prompt/confirm`는 사용하지 않는다.

이유:
- Tauri 오버레이에서 기본 브라우저 prompt/confirm은 UX와 포커스 측면에서 불안정했다.

### 처리 완료

- 클릭 즉시 내부 상태 전이
- 카드 목록에서는 바로 숨김
- DB에서는 삭제하지 않고 상태만 변경

이유:
- 리포트와 이력에는 계속 필요하기 때문이다.

## 6. 리포트 생성 UX

리포트는 세션 탭에서만 생성한다.

### 현재 정책

- 자동 생성 X
- 수동 생성 O

### 사용자 플로우

1. 세션 종료
2. 형식 선택
3. `리포트 생성` 클릭
4. 생성 중 상태 표시
5. 완료 후 파일 경로 / 버전 / 상태 반영

### 설계 이유

- 현재 품질 수준에서는 사용자가 종료 후 한 번 더 확인하는 흐름이 더 안전하다.
- 종료와 생성 실패를 분리하면 디버깅과 사용자 이해가 쉬워진다.

## 7. 숨긴 요소와 이유

현재 UI에서 일부 내부 기능은 의도적으로 숨겼다.

### 숨긴 것

- raw 상태값
- bulk lifecycle 액션
- 삭제 버튼
- 별도 리포트 탭
- 개발용 텍스트 입력 패널

### 이유

- 사용자가 이해해야 할 개념 수를 줄이기 위해서
- 현재 MVP 단계에선 “단순한 흐름”이 더 중요하기 때문

## 8. 현재 UX 품질 이슈

### live 자막

- STT 품질에 따라 partial 흔들림이 남아 있다.
- stability 기반 렌더로 체감은 줄였지만 완전히 해결되지는 않았다.

### 이벤트

- 질문 외 인사이트는 실시간 노출하지 않는다.
- 이유는 precision과 UX 복잡도 때문이다.

### 리포트

- 생성 플로우는 단순해졌지만, 리포트 본문 품질은 계속 개선 대상이다.

## 9. 향후 확장 후보

- 리포트 생성 중 progress 표시 강화
- 최신 리포트 바로 열기
- transcript fold/unfold
- 이벤트 수정 시 근거 문장 함께 표시
- 세션 종료 후 리포트 생성 유도 메시지 개선

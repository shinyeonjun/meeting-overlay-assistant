# 오버레이 UI/UX 명세서

> 문서 목적: 회의 중 화면 위에 노출되는 오버레이의 레이아웃, 정보 우선순위, 인터랙션, 시각 규칙을 정의한다.

---

## 1. UI 목표

오버레이의 목표는 화려한 대시보드가 아니다.

- 회의 화면을 방해하지 않으면서
- 지금 중요한 정보만 드러내고
- 사용자가 메모 부담 없이 회의 흐름을 따라가게 돕는 것

핵심 원칙:

```text
많이 보여주는 UI가 아니라,
적게 보여주되 놓치면 안 되는 것을 보여주는 UI
```

---

## 2. 정보 구조

오버레이는 크게 세 영역으로 구성한다.

1. `캡션 박스` (하단 중앙)
   - 실시간 자막 (live-text)
   - 이전 발화 피드 (feed-line)
   - 이벤트 피드 (feed-event)
   - 화자 라벨 (live-speaker)

2. `FAB 버튼` (우하단)
   - 워크스페이스 토글
   - 드래그 이동 가능

3. `워크스페이스` (우측 사이드 패널)
   - 세션 탭: 세션 생성, 세션 정보, 현재 주제, 개발용 텍스트 입력
   - 이벤트 탭: 질문, 결정, 액션, 리스크 카드
   - 리포트 탭: 오디오 경로, 리포트 생성, 화자별 전사/이벤트

---

## 4. 카드 종류별 역할

### 4.1 질문 카드

목적:
- 아직 끝나지 않은 논점을 계속 드러낸다.

표시 요소:
- 질문 제목
- 상태
- 필요 시 짧은 답변 요약

시각 원칙:
- 가장 눈에 띄게 유지
- 해결 전까지 쉽게 사라지지 않음

### 4.2 결정 카드

목적:
- 회의 중 이미 확정된 사항을 명확히 고정한다.

표시 요소:
- 결정 문장
- 확정 상태

시각 원칙:
- 안정적이고 단단한 느낌
- 자주 변하지 않는 정보처럼 보여야 함

### 4.3 액션 카드

목적:
- 후속 실행 내용을 구조화해 보여준다.

표시 요소:
- 담당자
- 작업 내용
- 기한

시각 원칙:
- 가장 실행 지향적으로 보여야 함
- 정보가 분리되어 한눈에 읽혀야 함

### 4.4 리스크 카드

목적:
- 장애, 우려사항, 확인 필요 항목을 눈에 띄게 표시한다.

표시 요소:
- 리스크 제목
- 현재 상태
- 필요 시 영향 범위

시각 원칙:
- 주의 신호 역할
- 과도하게 많아지지 않게 제한

### 4.5 라이브 맥락 영역

목적:
- 지금 무슨 이야기가 오가는지 짧게 감을 주는 보조 패널

표시 요소:
- 최근 발화 1~2줄
- 화면 제목
- 키워드

시각 원칙:
- 카드보다 덜 강조
- 메인 UI가 아니라 보조 정보처럼 보여야 함

## 5. 노출 우선순위

### 항상 보일 것

- 현재 주제
- 미해결 질문
- 최신 결정사항
- 최신 액션 아이템

### 상황에 따라 보일 것

- 활성 리스크
- 최근 맥락

### 자동 접힘 대상

- 해결된 질문
- 오래된 컨텍스트
- 중복 리스크

---

## 6. 인터랙션 규칙

### 기본 원칙

- 회의 중에는 최소 조작 UI를 지향한다.
- 사용자가 많이 클릭하게 만들지 않는다.

### 현재 구현된 인터랙션

- 캡션 박스 접기/펼치기 (애니메이션 적용)
- 워크스페이스 열기/닫기 (FAB 토글, scale+opacity 애니메이션)
- 탭 전환 (세션/이벤트/리포트, tab-in 애니메이션 + accent indicator)
- 캡션 박스 드래그 이동 (헤더 드래그)
- 워크스페이스 드래그 이동 (헤더 드래그)
- FAB 버튼 드래그 이동 (5px 임계값 — 클릭/드래그 구분)
- 클릭 투과 (Win32 GetCursorPos 폴링, UI 영역 외 클릭은 하위 앱으로 전달)

### 다음 단계 인터랙션

- 카드 직접 편집
- 드래그 앤 드롭 재정렬
- 복잡한 필터 UI

## 7. 상태 표현 방식

### 질문

- `open`: 강한 강조
- `answered`: 약화된 강조
- `unresolved`: 회의 종료 후 문서에서 별도 표시

### 결정

- `candidate`: 점선 또는 낮은 강조
- `confirmed`: 강한 고정형 카드

### 액션 아이템

- `candidate`: 회색 계열 / 낮은 강조
- `confirmed`: 담당자, 기한 분리 노출
- `updated`: 변경 배지 표시 가능

### 리스크

- `active`: 경고 강조
- `monitoring`: 약한 경고
- `resolved`: 축소 노출

---

## 8. 시각 디자인 원칙

### 정보 밀도

- 한 화면에 카드가 너무 많아지면 안 된다.
- 기본 노출 카드 수는 제한한다.

권장:
- 질문 1~2개
- 결정 1개
- 액션 아이템 1~2개
- 리스크 0~1개

### 타이포그래피

- 빠르게 훑기 쉬운 계층 구조
- 카드 제목은 굵고 짧게
- 부가 정보는 작고 조용하게

### 컬러 원칙

- 질문: 중립 + 강조
- 결정: 안정 색상
- 액션: 실행 색상
- 리스크: 경고 색상
- 컨텍스트: 저강도 색상

중요:
- 색만으로 상태를 구분하지 않는다.
- 배지, 라벨, 아이콘 또는 형태 차이를 같이 사용한다.

### 모션 원칙

- 카드 생성: 짧고 분명한 등장
- 카드 업데이트: 과하지 않은 하이라이트
- 카드 종료: 부드러운 축소 또는 접힘

과도한 애니메이션은 금지한다.

---

## 9. 반응형 및 배치 정책

### 데스크톱 기본

- 우측 세로 카드 스택
- 상단 주제 바
- 보조 패널은 좁게 유지

### 작은 화면 또는 발표 환경

- 보조 맥락 영역 축소
- 카드 수 자동 제한
- 전체보기는 별도 패널로 이동

### always-on-top 정책

- 발표 / 회의 앱을 완전히 가리지 않도록 반투명 또는 여백을 둔다.
- 사용자가 위치를 옮길 수 있어야 한다.

---

## 10. 전체보기 화면 명세

전체보기는 회의 중 메인 오버레이와 다르게 아래 목적을 가진다.

- 누적 이벤트를 시간순으로 점검
- 해결된 질문과 종료된 리스크 확인
- 회의 후 문서화 전 상태 점검

포함 요소:

- 세션 정보
- 현재 주제 / 주제 이력
- 질문 목록
- 결정사항 목록
- 액션 아이템 목록
- 리스크 목록
- 문서 생성 상태

---

## 11. 접근성 및 사용성 기준

- 텍스트 대비가 충분해야 한다.
- 작은 글씨에 과도하게 의존하지 않는다.
- 색약 사용자도 상태를 구분할 수 있어야 한다.
- 키보드로 최소 조작이 가능해야 한다.

---

## 12. 실패 상황 UX

### STT 지연 또는 입력 끊김

- UI는 멈춘 것처럼 보이면 안 된다.
- “입력 대기 중” 또는 “분석 중” 상태를 조용하게 표시한다.

### 이벤트 확신도 낮음

- 확정처럼 보이지 않게 `후보` 상태를 명확히 표시한다.

### 문서 생성 중

- 회의 종료 후에는 “문서 정리 중” 상태를 보여준다.

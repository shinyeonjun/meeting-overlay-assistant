# 문서 스타일 가이드

이 문서는 이 저장소에서 Markdown 문서를 일관되게 작성하기 위한 기준이다.

목표는 두 가지다.

1. 처음 보는 사람이 빠르게 이해할 수 있게 한다.
2. 시간이 지나도 어디에 무엇이 있는지 헷갈리지 않게 한다.

잘 쓴 GitHub 문서의 핵심은 문장을 화려하게 쓰는 것이 아니라, 문서의 역할과 독자를 명확히 나누는 것이다.

## 1. 문서 역할 먼저 정하기

문서를 쓰기 전에 먼저 이 질문부터 한다.

- 이 문서는 누가 읽는가
- 이 문서는 무엇을 결정하거나 설명하는가
- 이 문서는 어디에 있어야 하는가

경로별 기본 역할은 다음과 같다.

- 루트 `README.md`: 프로젝트 입구 문서
- `docs/README.md`: 문서 포털
- `docs/architecture/`: 현재 구조와 기술 경계
- `docs/product/`: 정책, UX, 기대 동작
- `docs/internal/`: 계획, 메모, 작업 기록
- `docs/research/`: 조사와 비교 근거

역할이 섞이면 문서가 금방 읽기 어려워진다.

## 2. 기본 작성 원칙

### 짧게 시작하고, 뒤에서 자세히 설명한다

좋은 문서는 첫 10줄 안에 이 문서가 왜 필요한지 드러난다.

권장 순서:

1. 한 줄 요약
2. 이 문서가 다루는 범위
3. 본문 세부 내용

### 제목은 검색 가능하게 쓴다

좋은 제목:

- `구조 승격 이행 가이드`
- `오버레이 UI/UX 명세`
- `이벤트정책 / 상태전이`

나쁜 제목:

- `정리`
- `메모`
- `생각`
- `기타`

### 문장은 선언형으로 쓴다

권장:

- `현재 공식 서버 기준 경로는 server/ 이다.`
- `frontend/ 는 레거시 참조본으로 유지한다.`

비권장:

- `아마 이쪽을 기준으로 보면 될 것 같다.`
- `일단 이렇게 해두자.`

### 독자 기준으로 쓴다

문서 작성자는 맥락을 이미 알고 있지만, 독자는 모른다.

그래서 아래를 항상 넣는다.

- 배경
- 현재 기준
- 예외 또는 레거시 여부
- 다음에 어디를 보면 되는지

## 3. README 작성 규칙

루트 README는 프로젝트 소개 페이지처럼 써야 한다.

반드시 들어가면 좋은 항목:

1. 프로젝트 한 줄 설명
2. 현재 공식 기준 경로 또는 구조
3. 빠른 시작
4. 문서 진입 링크
5. 현재 브랜치나 구조 이행 상태에 대한 짧은 설명

루트 README에 넣지 않는 것이 좋은 항목:

- 너무 상세한 설계 판단
- 장문의 작업 일지
- 내부 TODO 나 메모
- 긴 실험 기록

이런 내용은 `docs/` 아래로 뺀다.

## 4. 상세 문서 작성 규칙

아키텍처, 정책, 운영 문서는 아래 흐름을 권장한다.

1. 문서 목적
2. 현재 기준
3. 세부 구조 또는 규칙
4. 예외/주의사항
5. 관련 문서 링크

이 순서를 지키면 문서를 처음 읽는 사람도 길을 잃지 않는다.

## 5. 좋은 Markdown 습관

### 리스트는 평평하게 쓴다

한 문단 안에 너무 많은 중첩 리스트를 만들지 않는다. 문서가 갑자기 설계서가 아니라 미로처럼 보이기 쉽다.

### 코드 블록은 실제로 실행 가능한 예시를 우선한다

예시는 가급적 바로 복사해서 실행할 수 있는 형태가 좋다.

```powershell
uvicorn server.app.main:app --host 127.0.0.1 --port 8011
```

### 표는 비교가 필요한 곳에만 쓴다

표는 보기 좋아 보이지만 남용하면 수정 비용이 커진다. 비교 항목이 명확할 때만 쓴다.

## 6. 톤 가이드

이 저장소 문서는 아래 톤을 유지한다.

- 단정적이고 간결한 문장
- 한국어 자연체
- 영어 기술 용어는 원어 그대로 사용
- 과장된 홍보 문구보다 실제 동작 기준 우선
- 추측보다 현재 코드 기준 우선

## 7. 문서 수정 체크리스트

문서를 고치고 나면 아래를 확인한다.

- 이 문서는 누가 읽는지 분명한가
- 현재 기준 경로와 용어가 맞는가
- 레거시와 공식 기준을 구분했는가
- 다른 문서와 역할이 겹치지 않는가
- 다음에 읽어야 할 문서 링크가 있는가

## 8. 지금 이 저장소에서 특히 중요한 것

현재는 구조 승격 중이라서 아래 표현을 섞어 쓰지 않는 것이 중요하다.

- `server/client` = 공식 기준 경로
- `backend/frontend` = 레거시 참조본

문서가 이걸 흐리면 코드보다 문서가 더 큰 혼란을 만든다.

## 1. PostgreSQL 런타임 스키마 타입 정리

### 문제

초기 런타임 스키마는 PostgreSQL을 쓰고 있지만 일부 컬럼 계약이 SQLite 호환 시절의 관성에 남아 있었다.

대표 문제는 아래와 같았다.

- ID가 실제로는 UUID 계약이어야 하는데 문자열처럼 다뤄지는 구간이 있었다.
- 시간 컬럼은 PostgreSQL 기준으로 `TIMESTAMPTZ`가 더 적합했다.
- 로그인 ID, 이메일처럼 대소문자 무시 비교가 필요한 값은 `CITEXT`가 맞았다.
- runtime-compatible schema와 typed target schema가 섞이면 “지금 실제 DB가 어떤 기준인지” 판단하기 어려웠다.

### 해결

`020_runtime_with_pgvector_schema.sql`은 기존 runtime-compatible 기준으로 유지하고, typed target 기준을 별도 파일로 분리했다.

주요 파일:

- `server/app/infrastructure/persistence/postgresql/021_runtime_typed_target_schema.sql`
- `server/app/infrastructure/persistence/postgresql/022_runtime_typed_inplace_migration.sql`
- `server/app/infrastructure/persistence/postgresql/database.py`

정리 방향:

- fresh schema 기준안은 `021_runtime_typed_target_schema.sql`에 둔다.
- 실제 기존 DB에 덮어 적용하는 절차는 `022_runtime_typed_inplace_migration.sql`로 분리한다.
- UUID, `TIMESTAMPTZ`, `CITEXT`, `JSONB`, `vector` 기준으로 PostgreSQL 친화적인 계약을 잡는다.
- 실제 Docker PostgreSQL DB 적용은 백업 후 적용, 검증 후 백업 정리 흐름으로 처리했다.

### 판단

이 작업은 기능 최적화가 아니라 DB 계약 정리 작업이다. 이후 서버 코드와 테스트가 UUID 계약을 기준으로 안정적으로 움직이게 하는 기반 작업으로 본다.

### 남은 리스크

운영 DB에 적용할 때는 여전히 사람이 최종 승인해야 한다. DB 마이그레이션은 되돌리기 비용이 크므로, 개발 DB에서 통과했다고 운영 DB에 자동 적용하면 안 된다.

## 2. 런타임 자원 병목 분석과 구조 정리

### 문제

처음에는 CPU나 RAM이 병목일 가능성도 봤지만, 실제 측정 결과 가장 큰 병목은 `8GB VRAM dGPU`와 GPU-heavy 작업의 동시 실행이었다.

겹치면 위험한 작업:

- `faster-whisper` final STT
- `pyannote` diarization
- `Ollama gemma4:e4b` workspace summary

특히 `gemma4:e4b`는 실시간용이 아니고, 실행되면 메모리와 시간을 크게 사용한다.

### 해결

정책을 아래처럼 정리했다.

| 경로 | 처리 방향 |
| --- | --- |
| live mic | Web Speech API 우선, server GPU STT fallback 기본 off |
| live system audio | CPU partial + GPU final 구조 유지 |
| post-processing | 세션 종료 후 batch 처리 |
| diarization | 실시간이 아니라 post-processing 전용 |
| workspace summary | 비실시간 배치 작업 |
| worker 병렬도 | 무거운 worker를 많이 늘리지 않음 |

관련 문서:

- `docs/research/런타임_자원_병목_및_검증_정리.md`

### 검증

대표 검증 결과:

- 전체 서버 pytest: `562 passed, 1 skipped`
- overlay Node tests: `16 passed`
- overlay build: 통과
- web build: 통과
- 실제 15초 WAV E2E에서 post-processing, note correction, report generation 완료 확인

### 남은 리스크

첫 post-processing 처리 비용은 여전히 크다. cache가 같은 녹음 재처리에는 도움이 되지만, 처음 처리하는 `pyannote + faster-whisper` 비용 자체를 없애지는 않는다.

## 3. mic 경로의 GPU fallback 문제

### 문제

제품 관점에서 `mic`는 Web Speech API를 쓰는 것이 자연스럽다. 그런데 서버 쪽 `faster_whisper_streaming` fallback이 기본 연결되어 있으면, 사용자가 의도하지 않았는데도 mic가 GPU STT 경로로 들어갈 수 있다.

이 구조의 문제:

- mic 실시간 입력이 GPU를 점유한다.
- post-processing, summary, diarization과 GPU 경합이 생긴다.
- fallback인데 실제로는 기본 경로처럼 보일 수 있다.

### 해결

`mic`의 server STT fallback은 기본 off로 정리했다.

정책:

- 정상 제품 흐름: mic는 Web Speech API 중심
- 서버 fallback: 명시적으로 켠 예외 상황에서만 사용
- system audio는 서버 경로 유지

관련 검증에서 확인된 로그:

- mic audio WebSocket 연결 차단
- reason=`server_stt_fallback_disabled`

### 판단

이 결정으로 MVP 실시간 경로의 GPU 부담이 크게 줄었다. fallback은 성능 최적화 대상이 아니라 장애 대응 옵션으로 본다.

## 4. 실시간 질문 감지 범위 축소

### 문제

초기 live question 기능은 질문 감지뿐 아니라 아래까지 확장될 수 있었다.

- 질문 추가
- 질문 닫기
- 답변됨 판단
- 리스크 감지
- 액션 아이템 감지
- 결정 사항 감지

하지만 이 범위는 MVP 실시간 기능으로는 과했다.

문제점:

- 실시간 중 LLM 호출이 많아질 수 있다.
- STT 오류가 있으면 잘못된 상태 전이가 생긴다.
- 오버레이가 회의 보조보다 “분석 대시보드”처럼 복잡해진다.
- 사용자가 원한 핵심은 “방금 질문이 나왔다”를 놓치지 않는 것이다.

### 해결

MVP에서는 질문 감지만 남겼다.

현재 방향:

- deterministic candidate filter가 질문 후보를 넓게 잡는다.
- 작은 LLM은 후보가 실제 질문/확인 요청인지 판정한다.
- LLM output은 `add`만 허용한다.
- `close`, `answered`, `risk`, `action`, `decision`은 live MVP에서 제외한다.

주요 파일:

- `server/app/services/live_questions/question_lane_filter.py`
- `server/app/services/live_questions/question_prompt_builder.py`
- `server/app/services/live_questions/question_llm_client.py`
- `server/app/services/live_questions/question_dispatch_service.py`
- `client/overlay/src/controllers/live/live-feed.js`
- `client/overlay/src/controllers/live/live-payload-handler.js`

### 검증

관련 테스트:

- `tests/server/live_questions/test_question_lane_filter.py`
- `tests/server/live_questions/test_live_question_llm_client.py`
- `tests/server/live_questions/test_live_question_dispatch_service.py`
- `tests/server/live_questions/test_live_question_runtime_flow.py`
- overlay Node tests `16 passed`

### 남은 리스크

질문 감지는 결국 STT 품질에 묶인다.

예를 들어 STT가 `언제`, `왜`, `어떻게`, `되나요` 같은 의미 신호를 망치면 후보 필터도 놓칠 수 있다. 그래서 질문 감지는 “완벽한 판정기”가 아니라 “놓치지 않도록 보조하는 알림”으로 보는 게 맞다.

## 5. live 이벤트, 리스크, 액션 자동 생성 제거

### 문제

이전 테스트와 일부 흐름은 live text WebSocket 입력이 meeting event를 자동 생성한다고 기대했다.

하지만 MVP 정책을 질문-only로 줄이면서 live text가 자동으로 `overlay_events`를 만들면 정책과 충돌한다.

실제로 전체 pytest에서 아래 테스트들이 실패했다.

- `tests/server/api/test_events_api.py`의 이벤트 목록/필터/수정/삭제 테스트 5개

실패 원인:

- 테스트는 live text 입력 후 question event가 생성된다고 기대했다.
- 현재 MVP 정책에서는 live analyzer가 event를 자동 생성하지 않는다.
- 따라서 기능 파손이 아니라 테스트 계약 불일치였다.

### 해결

테스트 계약을 새 정책에 맞게 바꿨다.

변경 내용:

- live text WebSocket은 MVP에서 events를 자동 생성하지 않는다고 검증한다.
- 이벤트 조회/수정/삭제 API 검증은 DB에 이벤트를 직접 seed해서 검증한다.
- API CRUD 검증과 live analyzer 정책 검증을 분리했다.

수정 파일:

- `tests/server/api/test_events_api.py`

### 검증

검증 결과:

- `tests/server/api/test_events_api.py`: `5 passed`
- 전체 서버 pytest: `562 passed, 1 skipped`

## 6. AI 사용 과다 구조 정리

### 문제

프로젝트 곳곳에서 AI를 붙일 수 있는 구조였고, 그대로 두면 아래 문제가 생긴다.

- 실시간 경로가 무거워진다.
- LLM 호출 비용과 지연이 늘어난다.
- 단순 포맷팅이나 분류까지 AI가 맡아 결과가 흔들린다.
- 작은 로컬 개발 머신에서 GPU/VRAM 경합이 심해진다.

### 판단

AI가 필요한 곳과 코드가 더 나은 곳을 분리했다.

AI가 필요한 쪽:

- 회의 내용 요약
- 회의록 섹션 내용 추출
- 노트 인사이트 고도화
- 추후 챗봇/RAG

코드가 더 나은 쪽:

- PDF/HTML 템플릿 레이아웃
- 이벤트 상태 전이
- 단순 filtering
- artifact 저장 경로
- report section 고정 구조
- live MVP feature flag

### 해결

MVP에서는 아래 방향으로 정리했다.

- live risk/action/decision 분석 제거
- live question은 질문 후보 + 작은 판정으로 축소
- report 문서 모양은 AI가 아니라 HTML/CSS 템플릿으로 고정
- 노트 분석은 기본적으로 rule/code 기반을 우선하고, 고급 인사이트는 후순위

### 남은 리스크

회의록 본문을 얼마나 AI가 채울지는 아직 최종 확정 전이다. 한 번에 전체 회의록을 생성할지, 섹션별로 순차 생성할지는 실제 회의 샘플로 비교해야 한다.

## 7. post-processing stage 분리와 cache

### 문제

post-processing은 하나의 큰 덩어리 작업처럼 보였다.

구조상 무거운 단계:

- 녹음 파일 로드
- diarization
- final STT
- canonical transcript/event build
- persist
- note/report follow-up enqueue

처음 처리할 때는 `pyannote + faster-whisper` 비용이 크고, 어디서 시간이 쓰이는지 알기 어렵다.

### 해결

stage를 분리하고, 같은 녹음 재처리에 cache를 활용하는 방향으로 정리했다.

로그 기준 stage:

- `prepare`
- `load_audio`
- `diarize`
- `stt`
- `build`
- `persist`

cache hit이 가능한 부분:

- diarize stage
- stt stage

관련 파일:

- `server/app/services/post_meeting/session_post_processing_job_service.py`
- `server/app/services/post_meeting/post_processing_stage_cache.py`
- `tests/server/sessions/test_session_post_processing_job_service.py`

### 검증

전체 pytest 로그에서 stage별 처리와 cache hit을 확인했다.

대표 로그:

- `session post-processing diarize cache hit`
- `session post-processing stt cache hit`
- `session post-processing 전체 완료`

### 남은 리스크

cache는 재처리 최적화다. 첫 처리 자체를 빠르게 하려면 모델/하드웨어/worker 실행 정책을 추가로 조정해야 한다.

## 8. 노트/리포트 E2E 스크립트 보강

### 문제

`server/scripts/audio/generate_note_from_fixture.py`는 이름상 fixture WAV 하나로 노트/리포트까지 만드는 개발용 E2E 스크립트처럼 보였다.

하지만 실제 실행해보니 post-processing 뒤에 note correction 단계를 제대로 처리하지 못해 report job이 이어지지 않는 문제가 있었다.

문제 흐름:

1. 세션 생성
2. WAV 변환
3. post-processing 실행
4. note correction job이 남아 있음
5. report generation은 실제로 생성되지 않음

추가로 처음 수정 중 `NoteCorrectionJobService`에 없는 `process_latest_pending_for_session`를 호출해 실패했다.

### 해결

스크립트를 실제 pipeline 순서에 맞췄다.

현재 흐름:

1. session draft 생성
2. session start
3. fixture WAV를 runtime WAV로 변환
4. session finalize
5. session post-processing job 처리
6. latest note correction job 조회
7. note correction job 처리
8. report generation job 처리
9. final status와 artifact 경로 출력

수정 파일:

- `server/scripts/audio/generate_note_from_fixture.py`

### 검증

실제 15초 WAV로 검증했다.

명령:

```powershell
$env:PYTHONIOENCODING='utf-8'
venv\Scripts\python.exe server\scripts\audio\generate_note_from_fixture.py --wav tests\fixtures\video\test_16k_mono_15s.wav --title "MVP E2E note report script check"
```

결과:

- `post_processing_job.status`: `completed`
- `note_correction_job.status`: `completed`
- `report_job.status`: `completed`
- `final_report_status.status`: `completed`
- PDF artifact 생성 확인
- Markdown artifact 생성 확인
- analysis/transcript artifact 생성 확인

### 남은 리스크

`latest_report_preview`는 최신 report가 PDF일 때 `null`로 나온다. 기능 실패는 아니지만, 개발자 경험 관점에서는 Markdown preview를 따로 찾아 보여주도록 개선할 수 있다.

## 9. 회의록 고정 템플릿 방향

### 문제

기존 Markdown/PDF 리포트는 기능적으로는 생성되지만, 제품 회의록처럼 보이기에는 부족하다.

현재 부족한 점:

- 회의록 레이아웃이 고정된 문서 양식처럼 보이지 않는다.
- 사용자가 웹에서 입력할 참석자, 회의 장소, 회의 안건 같은 메타데이터 매핑이 아직 약하다.
- AI가 문서 구조까지 만들면 결과가 흔들릴 수 있다.

### 해결 방향

회의록은 고정 HTML/CSS 템플릿 기반으로 간다.

원칙:

- 템플릿은 repo에서 버전 관리한다.
- 생성된 PDF/Markdown/HTML은 artifact로 저장한다.
- AI는 문서 모양을 만들지 않는다.
- AI 또는 rule 기반 분석 결과는 정해진 섹션에만 들어간다.
- 참석자, 회의 장소, 회의 안건처럼 시스템이 모르는 값은 웹 입력값으로 받는다.

관련 파일:

- `temp/report_template_preview_v1.html`
- `server/app/services/reports/composition/html_report_template.py`
- `server/app/services/reports/composition/templates/report_v1/template.html`
- `server/app/services/reports/composition/templates/report_v1/styles.css`
- `tests/server/reports/test_html_report_template.py`

### 현재 상태

HTML 템플릿 렌더러와 테스트는 존재한다.

다만 아직 최종 제품 상태는 아니다.

남은 작업:

- 실제 세션 데이터에서 `ReportDocumentV1`을 만드는 mapper 추가
- 리포트 생성 시 HTML artifact 저장
- Markdown/PDF도 같은 source document에서 생성되도록 정리
- 웹 입력 메타데이터와 report document 연결
- 실제 회의 샘플로 PDF 레이아웃 검수

## 현재 해결된 것과 남은 것

### 해결된 것

- PostgreSQL typed schema 기준 분리
- 실제 DB 적용용 migration 흐름 정리
- GPU-heavy 작업이 겹치는 구조 문제 파악
- mic server STT fallback 기본 off
- live question 범위 축소
- live risk/action/decision 자동 분석 제거
- AI 사용 위치 정리
- post-processing stage 분리와 cache
- fixture WAV 기반 note/report E2E 스크립트 보강
- 전체 서버/오버레이/web 자동 검증 녹색 확인

### 남은 것

- 회의록 고정 템플릿에 실제 세션 데이터 매핑
- HTML artifact 저장
- PDF 렌더링을 템플릿 기반으로 더 가깝게 맞추기
- 웹에서 입력받을 회의 메타데이터 설계
- 실제 45초/7분 샘플로 회의록 품질 확인
- 질문 감지 품질을 실제 STT 결과 기준으로 평가
- `gemma4:e4b`를 어떤 report/summary 단계에 어느 정도 쓸지 최종 결정
- NPU/AMD Whisper는 아직 실험 경로로 유지

## 다음 작업 권장 순서

1. `ReportDocumentV1` mapper를 만든다.
2. report generation 시 HTML artifact를 같이 저장한다.
3. 실제 세션 데이터가 템플릿 섹션에 들어가는지 테스트한다.
4. 45초 WAV fixture로 빠른 E2E를 돌린다.
5. 7분 WAV fixture로 품질과 처리 시간을 확인한다.
6. 웹 입력 메타데이터를 회의록에 연결한다.

## 한 줄 요약

현재 CAPS MVP는 “라이브는 가볍게, 무거운 분석은 세션 종료 후, 회의록은 고정 템플릿에 데이터만 채우는 구조”로 정리됐다. 자동화된 기능 체인은 녹색이고, 다음 병목은 기능 동작이 아니라 실제 제품처럼 보이는 회의록 데이터 매핑과 출력 품질이다.
